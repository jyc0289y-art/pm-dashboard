# 품질관리 지침 (Quality Assurance)

> 산출물의 품질을 **사용자 검수 이전에 자체 검증**하는 원칙과 방법론.
> 모든 프로젝트(코드, 문서, 영상, 오디오, 일러스트, 데이터)에 공통 적용.

## 핵심 원칙

### 원칙 1: 검수 요청 전 자체 검증 필수
**사용자 검수는 최종 수락 단계이지, 품질 발견 단계가 아니다.**
- ❌ 산출물을 내놓고 "확인해 주세요"로 마무리
- ✅ "자체 검사 결과 A/B/C 확인, 발견 이슈 X건 수정 완료, 남은 Known Issue는 Y" → 그 후 검수 요청
- 자체 검사를 생략하고 사용자 검수를 요청하는 것은 사용자 시간을 훔치는 행위

### 원칙 2: 회귀 인지 (Regression Awareness)
**"작동한다"가 아니라 "이전보다 나아졌는가"가 품질의 정의.**
- 모든 산출물 갱신 시 **개선점 / 유지점 / 퇴보점**을 명시적으로 분리해 보고
- 퇴보가 발견되면 원인 분석 + 의도적 트레이드오프인지 버그인지 판단
- "새 기능만 자랑"은 퇴보를 숨긴다 — 변경되지 않은 영역도 능동 검증

### 원칙 3: 객관 지표 + 주관 지각 병행
**둘 중 하나만으로는 품질을 판정할 수 없다.**
- 객관: 파일 무결성, 스펙 준수, 성능 수치, 테스트 통과율
- 주관: 지각 품질, 가독성, 사용성, 정서적 반응
- 객관이 통과해도 주관이 실패할 수 있고 (예: 싱크 OK지만 리듬 엉망), 그 반대도 성립

### 원칙 4: 계층적 검사 (Layered QA)
**하위 계층이 실패하면 상위 계층 검사는 무의미하다.**
아래 순서로 검사하고, 실패 시 해당 층에서 즉시 중단:

```
Layer 1: 기술적 스펙 (Technical)
    ↓ 통과
Layer 2: 무결성 (Integrity)
    ↓ 통과
Layer 3: 지각 품질 (Perceptual)
    ↓ 통과
Layer 4: 서사/의미 (Narrative)
    ↓ 통과
Layer 5: 회귀 비교 (Regression)
    ↓ 통과
Layer 6: 사용자 수락 (Acceptance) ← 여기가 사용자 검수
```

## 이슈 심각도 분류

ISTQB + 커스텀 기준:

| 등급 | 명칭 | 정의 | 처리 |
|------|------|------|------|
| **P0** | Critical | 산출물이 목적을 달성하지 못함. 재생/실행/이해 불가 | 즉시 수정, 차단 이슈 |
| **P1** | Major | 주요 기능/품질 손상. 사용자가 명백히 인지 | 검수 전 수정 |
| **P2** | Minor | 사소한 결함. 인지되나 허용 가능 | 검수 시 공유, 후속 처리 |
| **P3** | Nit | 개선 제안, 선호 문제 | 백로그 등록 |

**보고 형식**: `[P0] 증상 — 원인 — 수정안`

## 계층별 검사 가이드

### Layer 1: 기술적 스펙 (Technical QA)

**목적**: 산출물이 기술 요구사항(포맷, 해상도, 인코딩, 사이즈 등)을 충족하는가.

**공통 체크 항목**:
- 파일 포맷/확장자 일치
- 스펙 준수 (해상도, 비트레이트, 샘플레이트, 코덱, FPS 등)
- 인코딩 무결성 (손상된 파일, 잘린 스트림 없음)
- 파일 크기 합리성 (예상 범위 내)
- 메타데이터 정확성 (타임스탬프, 인코딩 프로필)

**영상 특화 표준**:
- 해상도: 목표값과 일치 (1920×1080, 3840×2160 등)
- 프레임레이트: 일관성 (가변 FPS 여부 확인)
- 오디오 라우드니스 (EBU R128 / ITU-R BS.1770):
  - 방송/상업: -23 LUFS ±1
  - YouTube: -14 LUFS (목표) 
  - Podcast: -16 LUFS
- 색 공간: BT.709 (HD), BT.2020 (HDR)
- A/V 싱크: ±40ms 내 (ATSC 권고)

**검증 도구**:
- `ffprobe` — 스트림 정보
- `mediainfo` — 상세 메타데이터
- `ffmpeg -filter:a loudnorm=print_format=json` — 라우드니스 측정

---

### Layer 2: 무결성 (Integrity QA)

**목적**: 파이프라인 처리 과정에서 데이터가 손실되거나 왜곡되지 않았는가.

**공통 체크 항목**:
- 누락된 세그먼트/씬/파일
- 중복 / 이중 처리 흔적
- 경로/참조 끊김 (broken link, missing asset)
- 임시 파일이 최종 산출물에 포함
- 이전 버전 아티팩트가 섞임

**영상 특화**:
- 클립 총 길이 == 계획 길이 (±0.1초)
- concat 시 경계 프레임 깨짐 없음
- 오디오-비디오 런타임 일치
- 빈 프레임, 검은 프레임 (의도 외)
- 씬 경계에서 갑작스런 해상도/FPS 변화

**검증 방법**:
- 클립 합산 길이 = 롱폼 길이 (허용 오차 내)
- 각 씬/클립 duration 리스트업
- FFmpeg 에러 로그 grep (`error`, `invalid`, `warning`)

---

### Layer 3: 지각 품질 (Perceptual QA)

**목적**: 인간의 눈/귀로 인식되는 품질. 객관 수치가 통과해도 이 층에서 탈락 가능.

**시각 품질**:
- 해상도/샤프니스 (뭉개짐, 픽셀레이션)
- 색상 일관성 (씬 간 색온도/채도)
- 합성 품질 (알파 엣지, chromakey 스필, 그림자 불일치)
- 가독성 (텍스트 대비 WCAG 4.5:1 이상, 폰트 크기)
- 아티팩트 (밴딩, 블록 노이즈, 깜빡임)
- 구도 안정성 (카메라 흔들림, 의도 외 이동)

**오디오 품질**:
- 볼륨 밸런스 (화자 간, BGM vs 대사)
- 노이즈 / 클리핑 / 디스토션
- 대사 명료도 (masking, muffling)
- 싱크 (립싱크 ±80ms, BGM 큐 타이밍)
- 페이드 자연스러움 (갑작스런 컷오프)

**접근성 (WCAG 2.1)**:
- 텍스트 대비 비율 ≥ 4.5:1 (큰 텍스트 ≥ 3:1)
- 색상 단독 정보 전달 금지 (색맹 대응)
- 자막 가독성 + 동기화

**검증 방법**:
- 샘플 프레임 추출 (씬 중앙 + 전환점)
- 대표 프레임 시각 검수
- 오디오 파형 시각화 + 라우드니스 측정
- 대비 비율 계산기

---

### Layer 4: 서사/의미 (Narrative QA)

**목적**: 산출물이 전달하려는 의도/메시지/경험을 성취하는가.

**영상/스토리**:
- 이해 가능성: 처음 보는 사람이 이해할 수 있는가
- 감정 곡선: 계획된 감정 아크와 실제 결과 일치
- 리듬/페이싱: 정적-긴장-해소의 배분
- 훅(Hook) 효과: 첫 5~15초 몰입 유도
- 폐곡선: 프롤로그/에필로그의 주제적 연결

**문서/보고서**:
- 논리 흐름 (전제 → 근거 → 결론)
- 핵심 주장의 명확성
- 독자 관점 이해 가능성

**코드**:
- 의도가 코드에서 드러나는가 (self-documenting)
- 명명이 개념을 반영
- 구조가 도메인 모델과 일치

**검증 방법**:
- "이 산출물만 보고 의도를 재구성할 수 있는가?" 리트머스
- 원 기획안 대조 (Story Bible, 각본, 스펙)
- 외부인 관점 시뮬레이션

---

### Layer 5: 회귀 비교 (Regression QA)

**목적**: 이전 버전 대비 개선/유지/퇴보를 명시적으로 분리.

**비교 대상 선정**:
- 직전 버전 (점진적 개선 검증)
- 마지막 안정 버전 (누적 개선 검증)
- 같은 범주의 참조물 (벤치마크)

**비교 축**:
- 기술 스펙 (사이즈, 해상도, 런타임)
- 기능 범위 (커버된 씬/기능 수)
- 지각 품질 (시각/오디오 체크리스트)
- 성능 (빌드 시간, 리소스 사용량)
- 이슈 건수 (P0~P3 잔존 수)

**보고 형식 (필수)**:
```
┌─ 개선점 ────────────────────
│ • X 추가 → Y 해결
│ • A 수정 → B 향상
├─ 유지점 ────────────────────
│ • 기존 품질 유지 영역 (의도적)
├─ 퇴보점 ────────────────────
│ • C 부작용 (원인: D, 대응: E)
│ • F 누락 (의도적 삭제 / 실수)
└─ 트레이드오프 분석 ───────
   순이득/순손실 판정
```

**핵심**: 퇴보 0건을 보고할 때는 **능동적으로 찾았음**을 입증 (체크한 영역 목록 제시).

---

### Layer 6: 사용자 수락 (Acceptance)

**사전 조건**: Layer 1~5를 통과했거나, 미통과 항목을 Known Issue로 문서화했을 것.

**검수 요청 형식**:
```
[검수 요청] {산출물 이름}

자체 검사 결과
├─ Technical: ✅ (모든 스펙 충족)
├─ Integrity: ✅ (누락/중복 없음)
├─ Perceptual: ✅ (알려진 이슈 2건, 아래 참조)
├─ Narrative: ✅ (기획 의도 일치)
└─ Regression: ✅ (이전 대비 +X 개선, 퇴보 0건)

Known Issues (이번 검수 범위 외)
• [P2] ... (Phase Y에서 처리 예정)
• [P3] ... (백로그)

집중 검수 요청 포인트
• A 부분의 감정 전달이 의도대로인가?
• B의 타이밍이 자연스러운가?
```

## 프로젝트별 체크리스트 구축

각 프로젝트는 위 계층을 바탕으로 **도메인 특화 체크리스트**를 작성:
- 프로젝트 `.claude/CLAUDE.md`에 "품질 체크리스트" 섹션
- 자동화 가능한 항목은 스크립트화
- 반복되는 함정은 TRAP 시스템과 연동 (TRAP-NNN 역참조)

## 안티패턴

### 🚫 "빌드되면 됐지" 병
- 컴파일 성공 = 품질 보장 아님
- 파일 생성 성공 = 산출물 통과 아님
- 에러 없음 = 무결성 아님

### 🚫 사용자 눈을 QA로 쓰기
- 사용자에게 첫 재생/첫 읽기 시켜서 이슈 발견 → 사용자 시간 낭비
- 자체 검사 1시간 vs 사용자 왕복 1일 — 압도적으로 전자가 이득

### 🚫 체크리스트 암송
- "확인했습니다"만 적고 실제 확인 증거 없음
- 각 체크 항목에 **확인 방법 + 관찰 결과** 기록

### 🚫 개선만 자랑, 퇴보 숨김
- 회귀 검사 없이 신규 기능만 보고
- 묻지 않으면 말하지 않는 태도 = 신뢰 훼손

### 🚫 "의도된 동작입니다"로 도망
- 진짜 의도인지, 수정 비용 회피인지 구분 필요
- 의도적 트레이드오프라면 근거와 대안 제시
