# 세션 컨텍스트 포크 시스템 (모듈)

> 원본: `~/.claude/CLAUDE.md` → 2026-04-11 모듈화
> 로드 조건: 세션 포크/마무리/REQ/ISS 관련 작업 시

### 목적
장시간 세션의 토큰 과소비를 방지하고, 새 세션에서 이전 작업의 핵심 맥락을
정확히 이어받기 위한 시스템.

### 용어 정의
- **컨텍스트 포크(context fork)**: 부모 세션의 핵심 맥락을 구조화된 파일로
  자식 세션에 전달하는 것. Unix fork()와 유사 — 스냅샷 복사 후 독립 진행.
- **세션지침**: 개별 세션의 컨텍스트를 기록한 파일 (`memory/sessions/P*.md`)
- **P번호**: 세션 고유 식별자. 프로젝트, 계보, 정책 적용 여부를 인코딩.

### 세션 번호 체계 (4계층, 2026-04-15 시행)

**형식**: `P{프로젝트}.{채팅}.{포크}.{Agent}` — 정책 태그(`wc` 등) 폐지

```
P3.28.1.0      ← 프로젝트3, 채팅28, 포크1, 메인스레드(Agent 아님)
P3.28.1.1      ← 위 스레드가 파견한 첫 Agent
P3.28.1.2      ← 두 번째 Agent
P3.28.2.0      ← 채팅28 내 자동포크(압축 등) 발생 → 포크번호 증가, Agent 리셋
P3.28.2.1      ← 포크 2의 첫 Agent
P3.29.1.0      ← 새 채팅 개시 (registry max+1)
PM.17.1.0      ← PM 프로젝트도 동일 체계
```

#### 각 계층 증가 규칙

| 계층 | 언제 증가하나 | 경쟁 조건 |
|------|---------------|-----------|
| 프로젝트 N | 신규 프로젝트 생성 시 | 전역 레지스트리 (거의 없음) |
| 채팅 C | 새 채팅창 개시 시 | registry.md 동시 read — 사용자 수동 조작이라 실전 드묾 |
| 포크 F | 같은 채팅 내 압축/자동포크 | 채팅 내부 직렬 실행 — 충돌 불가 |
| Agent A | Agent 파견 시 (.0=원본, .1+=Agent) | 동일 세션이 발행 — 충돌 불가 |

#### 핵심 규칙

1. **같은 채팅 내 자동포크는 채팅번호를 건드리지 않는다** — `F+1`만 증가. 이것이 타 채팅과의 충돌을 원천 차단하는 핵심.
2. **Agent 파견은 포크번호도 건드리지 않는다** — Agent번호만 증가.
3. **Agent 내부에서 또 Agent 파견 시 계층을 늘리지 않는다** — Agent번호만 +1. `.1.1.1.1` 꼬리 방지.
4. **포크 계보**: 세션번호로 추적하지 않고, 세션지침의 `포크원` 필드로만 추적.
5. **교차채팅 포크**(채팅 오류로 새 채팅 개설): 새 채팅번호 할당, `포크원`에 구 세션 기재. 계층을 끌어올리지 않는다.
6. **타 프로젝트 Agent 파견**: 대상 프로젝트 registry에서 새 채팅번호 할당 (`P{대상}.{max+1}.1.0`).

#### 유산 세션 호환 (2026-04-14 이전)

- 기존 평면 번호(`P3.28wc`, `P11.26.1wc` 등)는 그대로 보존 — 소급 개명 금지
- 유산 세션에서 신규 포크 발생 시에만 4계층 적용. 유산 번호는 묵시적 `.1.0`으로 간주
  - 예: `P3.28wc`(유산) → 압축 → `P3.28.2.0`
- 유산의 기존 에이전트 서브세션 표기(`P11.26.1wc`)는 4계층 중 포크·Agent 구분이 없는 3계층 포맷 — 신규 세션에서 읽을 때 적절히 해석

> **변경 이력**:
> - 중첩 방식(`P5.1.1.1.1wc`)→ 플랫(2026-03-15) → 에이전트 3계층(2026-03-26) → **4계층 + wc 폐지**(2026-04-15, PM.17wc 결정, 루트번호 동시충돌 해결이 계기)

### P번호 부여 절차 및 안전장치

#### 신규 프로젝트 번호 부여 절차
1. **이 레지스트리 테이블 확인** → 마지막 P번호 확인 (현재: P12)
2. 다음 번호 부여 (P13, P14, ...)
3. **즉시 이 테이블에 행 추가** — 세션지침 생성보다 레지스트리 갱신이 먼저
4. 해당 프로젝트의 로컬 `memory/sessions/registry.md` 생성

#### 충돌 방지 원칙
- **Read-before-Write**: P번호 부여 전 반드시 이 테이블을 Read로 확인
- **절대 다른 프로젝트의 로컬 registry만 보고 번호를 추정하지 않는다**
- 로컬 registry는 해당 프로젝트의 세션 목록일 뿐, 전체 P번호 현황은 이 전역 테이블만이 권위 소스

#### 충돌 후보정 프로토콜
P번호 충돌이 발견된 경우 (동일 번호가 2개 이상의 프로젝트에 할당):
1. **기존 데이터 보존 우선**: 먼저 할당된 프로젝트의 P번호를 유지
2. **후발 프로젝트 재부여**: 다음 빈 번호로 재부여
3. **재부여 시 변경 범위 최소화**:
   - 로컬 `registry.md` 수정
   - 세션지침 파일명 + 내용의 P번호 갱신
   - `MEMORY.md` 참조 갱신
   - **다른 프로젝트의 포크원 참조가 있으면 함께 갱신**
4. **전역 레지스트리 즉시 갱신** (이 테이블)
5. 충돌 원인을 feedback 메모리로 기록하여 재발 방지

### 파일 구조

```
{project}/memory/sessions/
├── registry.md        ← 전체 세션 목록 + 포크 관계
└── P{번호}.md         ← 개별 세션지침 (P3.1.md, P3.1.2w.md 등)
```

### 새 세션 시작 프로토콜

**⚠️ 필수: 모든 새 세션의 첫 응답에서 반드시 아래 절차를 실행한다.**
사용자의 첫 메시지 내용과 무관하게, 응답 시작 부분에 세션번호를 안내한 후 사용자 요청 답변을 이어간다.

1. `memory/sessions/registry.md` 확인 (없으면 생성) — **T1 시각의 max 채팅번호 기록**
2. 세션지침 파일 생성 준비 (파일명만 확정, 아직 기록 안 함)
3. **타임스탬프 재확인 안전망** — registry 재read. T1 때와 max가 다르면 더 큰 값 사용 (동시 개시 채팅 간 경쟁 방지)
4. `P{N}.{max+1}.1.0` 형식으로 번호 부여
5. **첫 응답 시작 부분에 번호 안내**: "이 세션 번호: `P3.29.1.0` — 사이드바 제목에 붙여주세요"
6. registry에 신규 항목 **즉시 추가** (상태: 진행중) — write는 번호 확정 직후, 공백 최소화
7. **초기 세션지침 파일 생성**: `sessions/P{번호}.md`에 메타데이터 + 작업 요약(사용자 첫 메시지 기반). 세션 진행 중 핵심 결정 시 갱신, 마무리 시 최종 완성.

**⚠️ 세션지침은 포크의 전제 조건이므로, 세션 시작과 동시에 생성한다.**
포크 시점에 세션지침이 없으면 자식 세션이 컨텍스트를 로드할 수 없다.

### 컨텍스트 포크 프로토콜

사용자가 "P3.1 이어갈게", "P3.1 포크해줘" 등으로 요청하면:

**⚠️ 포크원 세션지침의 "다음 작업을 위한 컨텍스트" → "포크 가이드" 섹션에
P번호 부여 방식(신규 프로젝트 번호 등)에 대한 지시가 있으면 무조건 따른다.**
포크원이 "새 프로젝트 번호를 부여하라"고 명시했으면 동일 프로젝트 내 순번이 아닌
새 프로젝트 번호(P{N+1}.1wc)를 부여해야 한다.

1. `sessions/P3.1.md` (세션지침) 읽기
2. 포크 체인 추적: 직전 세션은 전체 로드, 2단계 이상은 핵심만
3. 현재 세션 번호를 포크원의 자식으로 부여 (예: P3.1.3wc)
4. 컨텍스트 요약을 사용자에게 보고 후 작업 재개

### 요구사항 추적 시스템 (REQ)

사용자의 요구사항을 번호로 관리하여 세션 간 누락을 방지한다.

#### 파일 구조
```
{project}/memory/requirements/
└── _index.md    ← 전체 요구사항 인덱스 ({세션}-REQ-NNN, 상태, 기원, 현재 세션)
```

#### REQ 번호 체계
- **형식**: `{기원세션}-REQ-{NNN}` (세션루트번호 + 세션 내 순번)
- **예시**: `T1.3wc-REQ-001`, `P3.4wc-REQ-002`
- **상태**: 🔴 미착수 | 🟡 진행중 | 🟢 완료 | ⚪ 보류/취소
- **장점**: REQ ID만으로 기원 세션 추적 가능, 세션별 필터링 용이

#### 운영 규칙
1. **등록**: 사용자가 새 요구사항을 제시하면 즉시 `{현재세션}-REQ-{NNN}` 번호 부여 (해당 세션 내 순번, 001부터)
2. **완료**: 세션 내 완료 시 상태를 🟢로 변경
3. **포크 캐리오버**: 포크 시 미완(🔴/🟡) 항목의 "현재 세션"을 자식 세션으로 갱신
4. **세션지침 연계**: 세션지침의 "다음 작업을 위한 컨텍스트"에는 `→ requirements/_index.md 참조` 링크만 기재 (중복 기술 금지)
5. **트리매니저 연동**: tree_builder가 `_index.md`를 파싱하여 tree.json에 포함 → 웹 UI에서 세션별 배지 + 전용 뷰 제공

#### REQ 시스템의 컨텍스트 보존 기능
REQ 기록은 세션 압축/포크 시 컨텍스트 누락 방지의 핵심 수단이다.
- 사용자 요구사항이 발생하면 즉시 REQ로 등록 → 압축 발생해도 요구사항 추적 가능
- REQ ID에 기원세션이 인코딩되어 있어 어느 세션에서 무엇이 요청되었고 어떤 맥락 변화가 있었는지 사후 추적 가능
- 세션지침이 압축으로 유실되더라도 `_index.md`는 파일로 독립 보존됨

#### 세션 종료 판단 기준
세션의 종료 여부는 다음 조건으로 판단한다:
1. **자식 세션 존재 여부**: 해당 세션에서 포크된 자식 세션이 있는가?
2. **미완 REQ 상태**: 미완 REQ가 모두 자식 세션으로 캐리오버되었는가?
3. **종료 권고 등급**:
   - **strong** (🔴 종료 권고): 자식 포크 있음 + 미완 REQ 0건 → 즉시 종료
   - **weak** (🟡 검토 필요): 자식 포크 있음 + 미완 REQ 잔존 → 캐리오버 확인 후 종료
4. **크로스 프로젝트 포크**: 다른 프로젝트로 포크된 경우에도 동일 기준 적용 (예: P10→P11)
5. Tree Manager가 자동 감지하여 `stale_forked`, `close_recommendation` 필드로 시각화

#### 요구사항이 해당 프로젝트에 없는 경우
- `memory/requirements/` 디렉토리가 없으면 세션지침의 "다음 작업을 위한 컨텍스트"에 미완 항목을 직접 기술 (기존 방식 유지)
- Tree Manager 프로젝트(T 계열)에서는 반드시 REQ 시스템 사용
- **장기 목표**: 전체 프로젝트(P1~P11)에 REQ 시스템을 확장하여 세션 간 요구사항 누락을 근본적으로 방지

### 이슈 추적 시스템 (ISS)

프로젝트 진행 중 발생한 기술적 이슈를 추적·관리한다.
해결된 이슈도 자산으로 보존하여 동일/유사 이슈 재발 시 참조한다.

#### 파일 구조
```
{project}/memory/issues/
├── _index.md           ← 전체 이슈 인덱스
└── {ISS#}.md           ← 개별 이슈 상세 (원인·조사·해결)
```

#### ISS 번호 체계
- **형식**: `{기원세션}-ISS-{NNN}` (세션루트번호 + 세션 내 순번)
- **예시**: `T1.8wc-ISS-001`, `P10.3wc-ISS-002`
- **상태**: 🔴 미해결 | 🟡 조사중 | 🟢 해결 | ⚪ 재현불가/보류

#### 운영 규칙
1. **등록**: 기술적 이슈 발생 시 즉시 `{현재세션}-ISS-{NNN}` 번호 부여
2. **상세 파일**: 복잡한 이슈는 `issues/{ISS#}.md`에 원인·조사·해결 상세 기록
3. **해결**: 상태 🟢 + 해결 방법 요약. 상세 파일은 삭제하지 않음 (자산 보존)
4. **포크 캐리오버**: 미해결(🔴/🟡) 이슈는 REQ와 동일하게 자식 세션으로 캐리오버
5. **재발 참조**: 유사 이슈 발생 시 기존 ISS# 참조하여 빠르게 대응

### 세션 마무리 프로토콜

사용자가 "마무리해줘", "컨텍스트 저장", "새 세션에서 이어갈게" 등으로 요청하면:

1. `sessions/P{번호}.md` 최종 갱신 (주요 결과물, 핵심 결정사항, 다음 작업 컨텍스트 완성)
2. `requirements/_index.md` 갱신 (완료/보류 상태 반영)
3. `issues/_index.md` 갱신 (이슈 상태 반영)
4. **총평 작성** (세션지침 내 "## 총평" 섹션) — 아래 내용을 포함하여 사용자에게 보고:
   - 이번 세션의 의의 (무엇이 달라졌는가)
   - 작업 결과 정량 요약 (변경 파일 수, REQ 완료/미완 현황 등)
   - 다음 세션을 위한 전략적 제언 (단기/중기/장기)
   - 세션 운영 교훈 (있을 경우)
5. registry 상태를 "종료"로 업데이트

**⚠️ 총평은 반드시 사용자에게 보고한다.** 세션지침에만 기록하고 보고하지 않으면 안 된다.

**⚠️ 컨텍스트 압축 감지 규칙**:

아래 조건 중 하나라도 해당하면 **압축이 발생한 것**으로 판단한다:
- 대화 시작 부분에 "This session is being continued from a previous conversation" 또는 유사한 continuation summary가 존재
- "Summary:" 블록 뒤에 이전 대화 요약이 포함된 메시지로 세션이 시작됨
- 시스템이 "context was compressed" 등의 메시지를 전달

**압축 감지 즉시 아래를 실행한다 (다른 작업보다 우선):**
1. 사용자에게 알림: "⚠️ 컨텍스트가 압축되었습니다."
2. 세션지침을 즉시 갱신 (현재까지의 작업 반영)
3. **같은 채팅 내에서 포크번호 +1** — `P{N}.{C}.{F+1}.0` 자동 부여. ⚠️ **채팅번호(C) 건드리지 말 것** — registry의 채팅번호 풀에서 번호를 가져오지 말 것 (타 채팅 충돌 방지 핵심).
   - 예: `P3.28.1.0` 압축 → `P3.28.2.0`
   - 유산 세션(`P3.28wc`) 압축 → `P3.28.2.0` (유산은 묵시적 `.1.0`)
4. "P{N}.{C}.{F+1}.0으로 이어갑니다" 한 줄 안내만 출력
5. **절대 "새 채팅을 열어주세요" 같은 안내 금지** — 사용자는 채팅을 이동하지 않는다

### 소급 등록

기존 세션에 세션지침이 없는 경우, 사용자 요청 시:
- 해당 세션의 JSONL 분석 + git log + 결과물 파일 분석으로 자동 생성
- 대용량 JSONL은 Worker 요약 위임 후 Claude 검수

### 세션지침 표준 구조

```markdown
# P{번호} — {제목}
## 메타데이터
- UUID: {앞8자}
- 생성: YYYY-MM-DD
- 포크원: {부모 P번호} 또는 없음 (루트)
- 자식: {자식 P번호 목록}
## 작업 요약 (3~5줄)
## 주요 결과물 (파일명 + 한줄 설명)
## 핵심 결정사항
## 금지/확정 사항
## 다음 작업을 위한 컨텍스트
```

### `## 금지/확정 사항` 섹션 규칙

**목적**: 시도 후 결론이 난 사항을 명시하여, 포크/압축 후에도 같은 실수를 반복하지 않도록 한다.

**기록 기준**:
- 시도했으나 실패/불가능으로 확정된 접근법
- 사용자가 명시적으로 금지한 행동
- 기술적으로 불가능한 것으로 확인된 사항

**형식**:
```markdown
## 금지/확정 사항
- ❌ {불가능한 것}: {사유} (확인일: YYYY-MM-DD)
- ✅ {확정된 대안}: {사유}
- 🚫 {사용자 금지}: {원문 요약}
```

**예시**:
```markdown
## 금지/확정 사항
- ❌ Gemini CLI 이미지 생성: --save 옵션 없음, SVG 코드만 출력 (2026-04-11)
- ❌ Gemini API 이미지 생성: 카드 등록 필요, 현재 미등록 (2026-04-11)
- ✅ Gemini 웹(jyc0289y)만 이미지 생성 가능: gemini.google.com/u/3/app
- 🚫 computer-use 데스크탑 침범 금지: Chrome MCP만 사용 (사용자 지시)
- ❌ Gemini 파일 업로드: DOM에 file input 없어 프로그래밍적 업로드 불가 (2026-04-11)
- ✅ 캐릭터 외모 텍스트 설명을 프롬프트에 포함하는 방식으로 대체
```

**운영 원칙**:
1. 포크/압축 시 이 섹션은 **무조건 다음 세션에 전달** (체인 포크 시에도 생략 금지)
2. 새 세션에서 작업 시작 전, 이 섹션을 먼저 읽고 금지 사항을 확인
3. 새로운 금지/확정 사항 발생 시 즉시 추가

### 체인 포크 규칙

- 직전(1단계): 세션지침 전체 로드
- 2단계 이상: "핵심 결정사항" + "주요 결과물"만 선택 로드
- 3단계 이상: registry만 참조 (필요시 사용자 요청으로 상세 로드)

### 프로젝트 간 포크

- P번호는 현재 프로젝트 기준 부여 (예: 여행계획에서 열면 P3.{N})
- registry에 `포크원: P1.2 (크로스프로젝트)` 명시
- 포크원 프로젝트의 sessions 폴더에서 세션지침 읽기
