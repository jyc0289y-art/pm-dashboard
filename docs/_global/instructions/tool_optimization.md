# 도구 결과 최적화

토큰 과소비의 주원인은 대형 도구 결과(특히 스크린샷)의 누적.
아래 규칙으로 도구 결과 크기를 최소화한다.

## 스크린샷 최소화 규칙

screenshot(~54K tok/건)은 최종 시각 확인에만 사용. 그 외는 텍스트 도구 우선:

| 확인 목적 | 사용 도구 | 토큰/건 |
|-----------|-----------|---------|
| 구조/텍스트/요소 | `preview_snapshot`, `read_page` | ~2K |
| CSS/스타일 값 | `preview_inspect` | ~0.5K |
| 에러 확인 | `preview_console_logs`, `preview_logs` | ~0.3K |
| API 응답 | `preview_network` | ~0.5K |
| 인터랙션 테스트 | `preview_click` → `preview_snapshot` | ~2.5K |
| **최종 시각 확인만** | `preview_screenshot`, `screenshot` | ~54K |

## 파일 읽기 최적화

- 전체 파일 대신 필요한 부분만 읽기 (Read의 `offset`/`limit` 활용)
- 대형 파일은 먼저 구조 파악 후 필요 섹션만 읽기
- 2000자 이상 도구 결과는 Worker 요약 위임 고려
- **10K tok 초과(약 2500줄+) 파일은 메인 컨텍스트에서 직접 읽지 말고 서브에이전트(Explore/general-purpose)에 요약 위임**. 대형 가이드·스펙 문서·트랜스크립트·로그가 대표 사례. 필요한 섹션만 offset/limit로 조회하거나, 전체가 필요하면 위임.

## 압축 감지 시 대응

자동 컨텍스트 압축이 발생하면:
1. 사용자에게 "압축 발생" 알림
2. 세션지침 저장 → 새 세션 포크 권고
3. 압축 전에 미리 세션을 분할하는 것이 최적
