## 1. 프로젝트 배경 및 목적

### 1.1 문제 정의

Ludlum Model 3030-2 Alpha-Beta Sample Counter의 기본 제공 소프트웨어(LMI Model 3030 Control Software v1.5.7)는 다음과 같은 한계가 있다:

- 데이터 로깅 시 raw count만 저장되며 Bq/CPM 환산값이 누락됨
- 이미 저장된 샘플에 코멘트를 추가/수정할 수 없음
- User-defined comment는 10자 제한, 측정 시작 전에만 입력 가능
- 장비에서 PC 연결 없이 측정하면 코멘트가 빈 채로 저장됨
- Get Samples를 수동으로 눌러야 데이터가 PC로 전송됨
- Background Subtract, Crosstalk Correction이 로그에 반영되는지 불명확

### 1.2 목표

RS-232 프로토콜을 이용한 커스텀 로깅 소프트웨어를 개발하여:

1. 측정 시작/완료 시 자동으로 데이터를 수신하고 로컬 DB에 저장
2. 시료명, 측정자, 측정 위치 등 자유로운 메타데이터를 측정 데이터에 연결
3. 장비 파라미터(효율, background, crosstalk)를 읽어와서 자동으로 Bq/CPM/DPM 환산
4. 측정 이력을 DB로 관리하고 CSV/Excel 내보내기 지원
5. (검증 후) 동일 측정 건에 대해 모드별 값을 크롤링

### 1.3 라이선스 검토 결과

- Ludlum은 제작사 매뉴얼(Section 7 "RS-232 Communication")에 통신 프로토콜, 명령어 전체, 응답 포맷을 공식 문서화하여 공개
- Software License Agreement는 LMI 제공 소프트웨어의 복제/배포에 대한 것이며, RS-232 프로토콜은 하드웨어 인터페이스 사양
- 공개된 프로토콜을 이용한 서드파티 소프트웨어 개발은 라이선스 위반에 해당하지 않음
- "Printer Mode" 기능의 존재 자체가 자사 소프트웨어 없이도 데이터 수신을 허용하는 설계 의도를 반영

## 2. RS-232 통신 프로토콜 사양

### 2.1 물리적 연결

- **인터페이스**: RS-232 (9-pin D-connector) → USB 변환 케이블 사용
- **Baud rate**: 9600
- **Data bits**: 8
- **Stop bits**: 1
- **Parity**: None
- **Flow control**: Hardware (RTS/CTS)

### 2.2 핀아웃 (장비 ↔ PC)

```
M3030       PC (9-Pin)
2 TXD  →    RXD 2
3 RXD  ←    TXD 3
5 GND  ─    GND 5
7 CTS  →    RTS 7
8 RTS  ←    CTS 8
```

### 2.3 프로토콜 규칙

- 명령 종료: LF (ASCII 10). CR+LF도 허용
- 응답 종료: CR+LF
- 명령은 대소문자 무관 (case insensitive)
- 미인식 명령 시 "ERROR-xx" 반환 (xx = 명령의 처음 2글자)

### 2.4 핵심 명령어 (전체 목록은 부록 A 참조)

#### 카운트 제어

| 명령 | 기능 | 비고 |
|------|------|------|
| C1 | 카운트 시작 | 전면 스위치 기준 시간 사용 |
| C2 | 진행 중 카운트 중지 | |
| C3 | 알람 클리어 | |
| C4 | 카운트 시작 + 완료 시 결과 자동 반환 | **핵심 명령** |
| C5 | 1분 카운트 + 결과 자동 반환 | |

#### 데이터 읽기

| 명령 | 기능 | 응답 포맷 |
|------|------|----------|
| RR | 현재 카운트 결과 | `HH:MM MM/DD/YY aaaaaa bbbbbb x tttt.t` |
| RS | 전체 저장 샘플 | `ssss MM/DD/YY HH:MM:SS aaaaaa bbbbbb ttttt.t x cccccccccc` (59바이트/건, $로 종료) |
| RE1 / RE2 | 알파/베타 효율 읽기 | 반환값 ÷ 10 = 실제 % |
| RB1 / RB2 | 알파/베타 Background | 반환값 ÷ 10 = 실제 cpm |
| RX1 / RX2 | Beta→Alpha / Alpha→Beta Crosstalk | 반환값 ÷ 10 = 실제 % |
| RV | 고전압 설정값 | nnnn (Vdc) |
| RT | 현재 카운트 시간 | nnnnn.n (초) |
| RU | 현재 코멘트 | 10자 문자열 |
| RA | 다음 샘플 번호 | nnnn (−1 = 총 저장 수) |
| F | 펌웨어 버전 | 문자열 |
| RD | 현재 날짜/시간 | HH:MM:SS MM/DD/YYYY |

#### 설정 변경

| 명령 | 기능 | 비고 |
|------|------|------|
| D0 | Scaler 모드 설정 | |
| D1 | CPM 모드 설정 | |
| D2 | Becquerel 모드 설정 | **3030-2 전용** |
| D3 | CPS 모드 설정 | **3030-2 전용** |
| D4 | 현재 모드 읽기 | 반환: SCA/CPM/BQ/CPS |
| SUxxxxxxxxxx | 코멘트 설정 (10자) | |
| L1 / L0 | 로깅 ON/OFF | |
| P1 / P0 | 프린터 모드 ON/OFF | |
| B0 / B1 / B2 | Background Subtract OFF/ON/상태 | |
| X0 / X1 / X2 | Crosstalk Correction OFF/ON/상태 | |

### 2.5 RR 응답 포맷 상세

```
HH:MM MM/DD/YY aaaaaa bbbbbb x tttt.t
```

- `aaaaaa`: 알파 카운트 (6자리)
- `bbbbbb`: 베타 카운트 (6자리)
- `x`: 모드 코드 — S(Scaler), C(CPM), B(Becquerel), P(CPS)
- `tttt.t`: 카운트 시간 (분). 카운트 진행 중이면 남은 시간

### 2.6 RS 응답 포맷 상세 (저장 샘플)

```
ssss MM/DD/YY HH:MM:SS aaaaaa bbbbbb ttttt.t x cccccccccc[CR][LF]
```

- 총 59바이트/건
- `ttttt.t`: 카운트 시간 (초 — RR의 분 단위와 다름, 주의)
- `cccccccccc`: 10자 코멘트
- 마지막에 `$[CR][LF]` (3바이트)로 종료

### 2.7 3030-2 전용 변경사항 (Appendix A)

- Count Mode에 DPM 대신 Becquerel(B)과 CPS(P) 추가
- D 명령: D2=Bq, D3=CPS, D4=읽기 (기본 3030의 D2=DPM과 다름)
- Sample Logging 타입 코드: S(Scaler), C(CPM), B(Becquerel), P(CPS)
- Bq/CPS 모드에서 LCD는 xxxx.xx 포맷 (소수점 2자리)
- OL(Overload) 표시가 LED 대신 LCD에 "OL" 텍스트로 표시됨

## 3. 구현 단계

### Phase 1: 프로토콜 검증 + 기본 로깅 (우선 구현)

#### 3.1 스니핑/검증 도구 (sniffer.py)

**목적**: 장비 응답 패턴을 실제로 확인하고 문서화

```
기능 요구사항:
1. 시리얼 포트 자동 탐색 (COM 포트 스캔)
2. 연결 파라미터: 9600/8/N/1, HW flow control
3. 대화형 명령 입력 → 응답 표시 (hex + ASCII 동시)
4. 응답 타임아웃 처리
5. 전체 세션 로그 파일 자동 저장 (timestamped)
```

**스니핑 시 반드시 검증할 항목**:

```
[검증 1] 기본 연결
  → F (펌웨어 버전 확인)
  → RD (날짜/시간 확인)
  → D4 (현재 모드 확인)

[검증 2] 모드별 RR 응답 차이 (핵심)
  → D0 (Scaler) → C4 → 응답 기록
  → (카운트 완료 후, 시료 그대로 둔 상태에서)
  → D1 (CPM) → RR → 응답 기록
  → D2 (Bq) → RR → 응답 기록
  → D3 (CPS) → RR → 응답 기록
  → 비교: 각 모드에서 aaaaaa/bbbbbb 값이 달라지는지 확인
  
[검증 3] 파라미터 읽기
  → RE1, RE2 (효율)
  → RB1, RB2 (background)
  → RX1, RX2 (crosstalk)
  → B2, X2 (background subtract / crosstalk 활성화 상태)
  → RV (고전압)

[검증 4] Background/Crosstalk 보정 반영 여부
  → B1 (background subtract ON) → 측정 → RR 응답의 값이 raw count인지 보정값인지
  → B0 (OFF) → 동일 시료 측정 → RR 비교
  → X1/X0 동일 테스트

[검증 5] Printer Mode 출력 포맷
  → P1 (프린터 ON) → 카운트 → 시리얼 포트로 자동 출력되는 데이터 포맷 캡처
  
[검증 6] C4 vs C1+RR 비교
  → C4의 자동 반환 포맷이 RR과 동일한지 확인
```

#### 3.2 기본 로깅 소프트웨어 (logger.py)

**기술 스택**:
- Python 3.10+
- pyserial (시리얼 통신)
- SQLite3 (로컬 DB)
- 표준 라이브러리만 사용 (외부 GUI 프레임워크 없음, CLI 우선)

**핵심 기능**:

```
1. 연결 관리
   - COM 포트 자동 탐색 (F 명령으로 핸드셰이크)
   - 연결 상태 모니터링, 자동 재연결
   - 장비 정보 표시 (펌웨어, 시리얼, 현재 설정)

2. 측정 워크플로우
   a) 측정 전: 시료 메타데이터 입력 (제한 없는 자유 텍스트)
      - 시료명 (필수)
      - 측정자
      - 측정 위치
      - 시료 유형 (스미어/필터/직접 등)
      - 비고 (자유 텍스트)
   b) 카운트 시작: C4 명령 전송
   c) 카운트 완료: 자동 수신된 결과 파싱
   d) 환산값 자동 계산 (효율/background/crosstalk 적용)
   e) DB에 raw data + 환산값 + 메타데이터 모두 저장
   f) 결과 즉시 표시

3. 장비 파라미터 자동 수집
   - 프로그램 시작 시 RE, RB, RX, RV, D4 읽기
   - QC Check 후 변경 가능성이 있으므로 수동 새로고침 명령 제공

4. 데이터 관리
   - 저장된 측정 이력 조회
   - 메타데이터 사후 편집 (시료명, 비고 등)
   - CSV 내보내기
   - 날짜/시료명/측정자 기준 필터링
```

#### 3.3 데이터베이스 스키마

```sql
-- 장비 파라미터 스냅샷 (측정 시점의 설정값 보존)
CREATE TABLE instrument_params (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    captured_at TEXT NOT NULL,           -- ISO 8601
    firmware_version TEXT,
    high_voltage_vdc INTEGER,
    alpha_efficiency_pct REAL,           -- 4π %
    beta_efficiency_pct REAL,
    alpha_background_cpm REAL,
    beta_background_cpm REAL,
    alpha_to_beta_crosstalk_pct REAL,
    beta_to_alpha_crosstalk_pct REAL,
    background_subtract_on INTEGER,      -- 0/1
    crosstalk_correction_on INTEGER,     -- 0/1
    count_mode TEXT                       -- SCA/CPM/BQ/CPS
);

-- 측정 데이터
CREATE TABLE measurements (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    
    -- 장비 원본 데이터
    instrument_datetime TEXT,             -- 장비 내부 시계 기준
    pc_datetime TEXT NOT NULL,            -- PC 시계 기준 (신뢰)
    alpha_raw_count INTEGER NOT NULL,
    beta_raw_count INTEGER NOT NULL,
    count_time_seconds REAL NOT NULL,
    count_mode TEXT NOT NULL,             -- S/C/B/P
    instrument_comment TEXT,              -- 장비 10자 코멘트
    
    -- 환산값 (자동 계산)
    alpha_cpm REAL,
    beta_cpm REAL,
    alpha_net_cpm REAL,                   -- background 차감
    beta_net_cpm REAL,
    alpha_bq REAL,                        -- 효율 적용
    beta_bq REAL,
    
    -- 파라미터 참조
    params_id INTEGER REFERENCES instrument_params(id),
    
    -- 사용자 메타데이터 (제한 없음)
    sample_name TEXT,
    operator TEXT,
    location TEXT,
    sample_type TEXT,
    notes TEXT,
    
    -- 관리
    created_at TEXT DEFAULT (datetime('now', 'localtime')),
    updated_at TEXT
);

-- 검색 인덱스
CREATE INDEX idx_measurements_datetime ON measurements(pc_datetime);
CREATE INDEX idx_measurements_sample ON measurements(sample_name);
CREATE INDEX idx_measurements_operator ON measurements(operator);
```

#### 3.4 환산 계산 로직

```python
# 기본 환산
alpha_cpm = alpha_raw_count / (count_time_seconds / 60)
beta_cpm = beta_raw_count / (count_time_seconds / 60)

# Background 차감 (net count rate)
alpha_net_cpm = alpha_cpm - alpha_background_cpm
beta_net_cpm = beta_cpm - beta_background_cpm

# Crosstalk 보정
# alpha_to_beta: 알파 카운트 중 베타 채널에 간섭되는 비율
# beta_to_alpha: 베타 카운트 중 알파 채널에 간섭되는 비율
alpha_corrected = alpha_net_cpm - (beta_net_cpm * beta_to_alpha_pct / 100)
beta_corrected = beta_net_cpm - (alpha_net_cpm * alpha_to_beta_pct / 100)

# Bq 환산 (2π geometry → 4π efficiency 사용)
# 장비 효율이 4π 기준이므로 직접 나누기
alpha_bq = alpha_corrected / (alpha_efficiency_pct / 100) / 60  # cpm → cps → Bq
beta_bq = beta_corrected / (beta_efficiency_pct / 100) / 60

# 참고: DPM = cpm / (efficiency / 100)
#        Bq = DPM / 60
```

**주의**: 위 계산은 기본 3030 매뉴얼의 DPM 계산 로직(Section 7)을 기반으로 한 것. Phase 1 스니핑에서 장비의 실제 Bq 표시값과 비교하여 검증 필요.

### Phase 2: 모드별 크롤링 (Phase 1 검증 후)

Phase 1 스니핑 결과에 따라 구현 여부 결정.

**가능한 경우 (RR이 모드별 환산값을 반환)**:
- 카운트 완료 후 D0→RR, D1→RR, D2→RR, D3→RR 자동 순회
- 4가지 모드의 값을 모두 한 레코드에 저장
- 장비 내부 계산값과 커스텀 계산값 비교 검증 가능

**불가능한 경우 (RR이 항상 raw count 반환)**:
- 모드 크롤링 불필요 (어차피 raw count에서 자체 계산)
- 대신 LCD 표시값(장비 Bq)과 자체 계산 Bq의 비교는 최초 1회 수동 검증으로 대체

## 7. 장비 사양 참조 (매뉴얼 발췌)

### 검출기 효율 (4π geometry, 제작사 공칭값)

| 핵종 | 효율 | 방사선 |
|------|------|--------|
| Th-230 | 32% | Alpha |
| U-238 | 39% | Alpha |
| Pu-239 | 37% | Alpha |
| C-14 | 8% | Beta |
| Tc-99 | 27% | Beta |
| Cs-137 | 29% | Beta |
| Sr-90/Y-90 | 26% | Beta |

### Background 기준 (10 µR/hr 환경)

- Alpha: 3 cpm 이하
- Beta: 50 cpm 이하

### MDA 공식 (매뉴얼 Section 7)

```
MDA(dpm) = (2.71 + CL × √(Background × Ts × (1 + Ts/Tb))) / (E × Ts)

Tb = Background Count Time (min)
Ts = Source Count Time (min)
E  = Counter Efficiency (fractional, not %)
CL = Confidence Level
     90% → 2.5632
     95% → 3.290
     99% → 4.6526
```

### DU 측정 관련 참고

- DU(열화우라늄)는 U-238이 99.7% 이상
- Alpha: U-238 (4.198 MeV) → 장비 효율 ~39% (4π)
- Beta: Th-234 (βmax 0.199 MeV) + Pa-234m (βmax 2.29 MeV)
- 방사평형 시 alpha/beta 모두 검출
- DU 효율 교정은 Pu-239(alpha)/Tc-99(beta) 표준선원 기반이므로, 에너지 차이에 의한 효율 보정이 필요할 수 있음
