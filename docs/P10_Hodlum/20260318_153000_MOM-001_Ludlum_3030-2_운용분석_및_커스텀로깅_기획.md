## 1. 회의 배경

DU(열화우라늄) 시료 측정을 위해 Ludlum Model 3030-2 Alpha-Beta Sample Counter를 운용하는 과정에서, 한국 대리점(동양이엔피테크) 제공 매뉴얼(7페이지 요약본)의 정보 부족으로 정상 운용이 어려운 문제가 발생. 제작사 원본 매뉴얼을 확보하여 장비 프로토콜을 분석하고, 기본 제공 소프트웨어의 한계를 극복하기 위한 커스텀 소프트웨어 개발을 기획.

---

## 2. 논의 내용

### 2.1 DU 측정 절차 정리

- 3030-2는 ZnS(Ag)+플라스틱 신틸레이터 기반 이중 채널 카운터로, DU의 알파(U-238)와 베타(Th-234, Pa-234m) 동시 측정에 적합
- 측정 전 반드시 QC Check 수행 필요
- Crosstalk Correction 활성화 권장 (알파/베타 동시 방출 특성)
- DU 표준선원으로 별도 효율 교정이 정밀 정량 분석의 정석
- 자기차폐(self-shielding) 문제: 두꺼운 시료보다 스미어/얇은 증착 시료가 유리

### 2.2 Background 보정 오해 바로잡기

- **현장 동료의 주장**: "DU 시료를 올려놓은 상태로 측정해도 background 보정이 된다"
- **사실**: 3030-2에는 셔터가 없으므로 시료 장착 상태에서 background 분리는 물리적으로 불가능
- **실제 동작**: "Activate Background Subtract"는 사전에 빈 트레이로 측정한 background 값을 산술적으로 차감하는 기능
- **결론**: 대리점 직원의 설명을 동료가 오해한 것으로 판단. QC Check 절차(Alpha → Beta → 빈 트레이 Background)가 이를 입증

### 2.3 데이터 로깅 문제 해결

**증상**: Log All 상태인데 소프트웨어에 데이터가 안 뜸

**원인 및 해결**:
- Data Logging 탭에서 **Get Samples 버튼**을 눌러야 장비 내부 메모리 → PC로 전송됨 (자동 아님)
- Logging Mode 변경 후 반드시 **Update 클릭** 필요
- Log All 설정 이전 측정분은 장비 메모리에 저장되지 않음

### 2.4 데이터 로깅 타입 코드 해명

**한국 대리점 매뉴얼에 누락되어 있던 정보**:

| 코드 | 한국 매뉴얼 (부정확) | 제작사 원본 (3030-2 Appendix A) |
|------|---------------------|-------------------------------|
| S | (미기재) | Scaler |
| C | (미기재) | Counts per minute (CPM) |
| B | (미기재) | **Becquerel** (3030-2 전용) |
| P | (미기재) | **Counts per second** (3030-2 전용) |
| D | (미기재) | DPM (기본 3030 전용, 3030-2에는 없음) |

- 진영이 단위를 변경하며 측정해본 결과 B=Becquerel임을 실험적으로 확인
- 한국 대리점 매뉴얼에는 타입 코드 정의가 완전히 누락되어 있었음

### 2.5 데이터 로깅의 근본적 한계

1. **raw count만 로깅됨**: LCD에 Bq로 표시되더라도 로그에는 raw count + count time만 저장. CPM/Bq 환산은 사용자가 직접 계산해야 함
2. **사후 코멘트 불가**: User-defined comment는 측정 시작 전에만 입력 가능, 10자 제한, 이미 저장된 데이터에 코멘트 추가/수정 불가
3. **PC 미연결 시 코멘트 빈 채로 저장**: 전면 COUNT 버튼만 눌러서 측정하면 코멘트 없이 저장
4. **날짜 소급 수정 불가**: 장비 시계 오류로 잘못 기록된 과거 데이터는 장비 내에서 수정 불가, CSV 내보내기 후 엑셀에서 수동 보정 필요

**raw count 로깅에 대한 판단**: 데이터 무결성 관점에서는 오히려 올바른 설계. 효율 재교정 시 과거 데이터 소급 재계산 가능. 다만 사용 편의성은 떨어짐.

### 2.6 Background/Crosstalk 보정이 로그에 반영되는지 여부

- 매뉴얼만으로는 확실하지 않음
- Data Logging 스크린샷에서 소수점이 포함된 값(24.6)이 존재하여, Count Mode에 따라 저장 값의 성격이 달라질 가능성 있음
- **확인 방법**: 보정 ON/OFF 상태에서 동일 시료 측정 후 로그 값 비교 (Phase 1 스니핑에 포함)

### 2.7 장비 효율 입력의 실제 용도

데이터 로깅과 무관하게 다음 기능에서 사용됨:
- LCD 실시간 Bq 표시 (현장 즉시 판단)
- 알람 임계값 판정
- QC Check 합격/불합격 판정
- MDA 계산

### 2.9 커스텀 로깅 소프트웨어 기획

#### RS-232 프로토콜 핵심

- 9600 baud, 8N1, HW flow control (RTS/CTS)
- LF로 명령 종료, CR+LF로 응답 종료
- **C4 명령**: 카운트 시작 + 완료 시 결과 자동 반환 (핵심)
- 전체 명령어 목록 확보 (Read/Set 각 20+ 개)

#### 모드별 크롤링 가능성 검토

- 카운트 완료 후 D0→RR, D1→RR, D2→RR, D3→RR 순회하여 모든 모드의 환산값 수집 가능 여부
- RR이 "현재 모드의 환산값"을 반환하는지 "항상 raw count"를 반환하는지가 핵심
- **Phase 1 스니핑에서 검증 예정**

#### 라이선스 검토

- RS-232 프로토콜은 하드웨어 인터페이스 사양이며 매뉴얼에 공식 문서화되어 공개됨
- Software License Agreement는 LMI 제공 소프트웨어에만 적용
- Printer Mode 기능의 존재가 서드파티 데이터 수신을 허용하는 설계 의도를 반영
- **결론: 커스텀 소프트웨어 개발은 라이선스 위반이 아님**
