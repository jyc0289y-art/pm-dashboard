## 1. 프로젝트 개요

### 1.1 목표

ITS(국가교통정보센터) CCTV 영상 및 공개 교통 데이터 API를 결합하여, 고속도로 상의 이상 상황을 실시간으로 감지하고 관제사에게 알림을 제공하는 AI 관제 시스템을 개발한다.

### 1.2 사업 구조

| 역할 | 주체 | 담당 |
|------|------|------|
| **개발사** | SL Corporation | AI 엔진 개발, 소프트웨어 설계/구현, 제품화 |
| **납품 파트너** | 파트너사 (기존 관급 영업) | 관급기관 영업, 입찰/수주, 납품, 현장 대응, GS인증/보안심사 |

- 파트너사는 기존에 관급기관 관제시스템을 납품하고 있는 회사로, 관급 사업 수주/납품 채널을 보유
- SL Corporation은 AI 엔진을 독자 제품으로도 운영하면서, 파트너사를 통해 관급기관에 모듈 납품

### 1.3 핵심 기능 요구사항

파트너사(관급 영업)의 원본 요구사항:

| # | 기능 | 설명 |
|---|------|------|
| F1 | 양방향 통행 인식 | 차량 추적을 통해 각 차선의 주행 방향을 식별 |
| F2 | 역주행 감지 | 정상 흐름 방향과 반대로 이동하는 차량 탐지 및 알림 |
| F3 | 보행자/이륜차 인지 | 고속도로 내 보행자, 자전거, 오토바이 감지 |
| F4 | 정체/사고 인지 | 교통 정체 판별 및 사고 발생 감지 |
| F5 | 악천후 대응 | 야간, 눈, 비, 안개 환경에서도 F1~F4 기능 유지 |
| F6 | 이벤트 알림 | 사고/역주행/보행자 감지 시 관제사에게 실시간 알림 |

### 1.4 제품 전략: 듀얼 모드

이왕 개발하는 만큼, **SL 단독 제품**으로도 성립하면서 **관급 시스템에 모듈로 통합**도 가능한 구조를 목표로 한다.

```
┌─────────────────────────────────────────────────┐
│              HighLink AI Engine                   │
│  (핵심 감지 엔진 — 두 모드 공통)                  │
│  YOLO11 + ByteTrack + 룰 엔진 + 데이터 퓨전      │
├─────────────────────────────────────────────────┤
│                    │                             │
│    ┌───────────────┘───────────────┐             │
│    ▼                               ▼             │
│  [모드 A: SL 독립 제품]     [모드 B: 모듈 납품]   │
│  자체 관제 UI + DB + 알림    gRPC/REST API만 노출  │
│  SL이 직접 운영/판매        파트너사 관제 시스템에  │
│                             임베딩               │
└─────────────────────────────────────────────────┘
```

- **모드 A (독립 제품)**: SL Corporation 자체 브랜드. 중소 관제센터, 민간, 해외 등
- **모드 B (모듈 납품)**: 파트너사를 통해 기존 관급 관제 시스템에 AI 엔진만 통합

> **설계 원칙**: 핵심 엔진은 하나. 위에 무엇을 얹느냐(자체 UI vs API만)로 모드가 갈린다.

### 1.5 비즈니스 컨텍스트

- **관급 고객**: 한국도로공사, 국토교통부, 지자체 교통정보센터, 지방경찰청
- **운영 환경**: 대부분 폐쇄망 또는 제한된 외부 접근. 온프레미스 배포 필수
- **기존 고객 시스템**: 대부분 Java/Spring 기반 SI 납품물. 데이터 교환은 REST (XML/JSON)
- **데이터**: ITS 오픈데이터 API를 통해 관제시스템 투입 영상과 동급의 CCTV 영상 확보 가능

## 2. 데이터 소스

### 2.1 ITS 오픈데이터 API (https://www.its.go.kr/opendata/)

인증키 발급 필요 (회원가입 → 마이페이지 → 인증키 신청).

| 데이터 | 용도 | 비고 |
|--------|------|------|
| **CCTV 영상** | AI 분석의 주 입력 | HLS(m3u8) 또는 MP4 스트림. 고속도로/국도 |
| **교통소통정보** | 구간별 실시간 속도. 정체 판단의 ground truth | VDS 기반 평균 주행속도 |
| **VDS 원시데이터** | 교통량, 점유율, 속도 (30초 주기) | 영상 분석 결과 교차검증용 |
| **돌발정보** | 사고/공사/기상 이벤트 | 미감지 사고 선제탐지 입증용 |
| **전국표준노드링크** | 도로망 네트워크 (노드+링크) | CCTV→도로 방향 자동 매핑에 필수 |

### 2.2 CCTV API 호출 예시

```python
import requests

API_KEY = "발급받은_인증키"
url = "https://openapi.its.go.kr:9443/cctvInfo"
params = {
    "apiKey": API_KEY,
    "type": "ex",        # ex: 고속도로, its: 국도
    "cctvType": 2,       # 1: 실시간스트리밍(HLS), 2: 동영상(MP4), 3: 정지영상
    "minX": 126.5,       # 경도 범위
    "maxX": 127.5,
    "minY": 36.0,        # 위도 범위
    "maxY": 37.5,
    "getType": "json"
}
response = requests.get(url, params=params)
cctv_list = response.json()["response"]["data"]

for cctv in cctv_list:
    print(f"이름: {cctv['cctvname']}")
    print(f"좌표: ({cctv['coordy']}, {cctv['coordx']})")
    print(f"영상URL: {cctv['cctvurl']}")
```

### 2.3 보조 외부 데이터

| 데이터 | 출처 | 용도 |
|--------|------|------|
| 기상 실황/예보 | 기상청 API (data.go.kr) | 안개/강우/적설 조건에 따른 감지 파라미터 동적 조정 |
| 도로 공사정보 | ITS 오픈데이터 | 오탐 필터링 (공사 구간 차량 정지 ≠ 사고) |

### 2.4 학습 데이터

- ITS 오픈데이터 CCTV 영상은 실제 관제시스템 투입 영상과 동급 → **학습 데이터로 직접 활용 가능**
- COCO pretrained 모델로 시작, ITS CCTV 데이터로 점진적 fine-tuning
- 사고/역주행 등 희소 이벤트: ITS 영상 장기 수집 + 공개 데이터셋(UA-DETRAC, BDD100K) 보완
- 날씨별/시간대별 데이터 균형 수집 필요
- **개인정보(번호판, 얼굴)**: 학습 데이터 내 비식별화 처리 필수. Phase 0에서 법률 검토 착수

## 3. 시스템 아키텍처

### 3.1 설계 원칙

1. **엔진과 UI를 분리**: 핵심 AI 엔진은 독립 프로세스. 위에 자체 UI를 붙이면 독립 제품, API만 노출하면 모듈 납품
2. **Phase별 점진 확장**: PoC는 최소 구성, 검증 후 확장
3. **표준 인터페이스**: 엔진 ↔ 외부 연동은 gRPC + REST. 관급 시스템(Java/Spring)에서 호출 용이

### 3.3 핵심 엔진 인터페이스 (모드 A/B 공통)

엔진이 외부에 노출하는 인터페이스:

```protobuf
// 이벤트 스트림 — 실시간 이벤트 수신
service HighLinkEngine {
  // 실시간 이벤트 스트림 (서버 → 클라이언트)
  rpc SubscribeEvents(EventFilter) returns (stream DetectionEvent);

  // 카메라 상태 조회
  rpc GetCameraStatus(CameraRequest) returns (CameraStatus);

  // 카메라 추가/제거
  rpc ManageCamera(CameraConfig) returns (Result);

  // 감지 파라미터 조정
  rpc UpdateDetectionConfig(DetectionConfig) returns (Result);
}

message DetectionEvent {
  string event_id = 1;
  string event_type = 2;       // wrong_way, pedestrian, congestion, accident
  string camera_id = 3;
  string camera_name = 4;
  double confidence = 5;
  string timestamp = 6;
  string severity = 7;         // critical, warning, info
  string message = 8;          // 템플릿 기반 한국어 메시지
  bytes thumbnail = 9;         // 이벤트 시점 썸네일
  map<string, string> meta = 10; // 추가 정보 (속도, 차선 등)
}
```

> **이 인터페이스가 모드 A와 모드 B의 경계.** 모드 A는 이 인터페이스 위에 자체 UI/DB/알림을 구축. 모드 B는 이 인터페이스를 파트너사에 그대로 제공.

### 3.4 Phase별 구성

| 구성 요소 | Phase 0-1 (PoC) | Phase 2 (MVP) | Phase 3 (제품) |
|-----------|-----------------|---------------|----------------|
| **추론 언어** | **Python** (ultralytics) | **Python→C++** (DeepStream) | **C++** (DeepStream/TensorRT) |
| **백엔드 언어** | **Python** (FastAPI) | **Java** (Spring Boot) 도입 | **Java** (Spring Boot/eGovFrame) |
| 영상 수집 | FFmpeg + Python | GStreamer/DeepStream | GStreamer/DeepStream (C++) |
| 감지 모델 | YOLO11 (PyTorch) | YOLO11 (ONNX/TensorRT) | YOLO11 (TensorRT) |
| 추적 | ByteTrack | ByteTrack | ByteTrack + BoT-SORT |
| 이벤트 감지 | 역주행 룰만 | 전체 룰 엔진 | 전체 + 퓨전 고도화 |
| 외부 API 연동 | CCTV만 | + VDS + 기상 + 공사 | + 돌발 + 노드링크 |
| 엔진 인터페이스 | REST (FastAPI) | gRPC (Java) + REST | gRPC + REST + WebSocket |
| 모드 A UI | 최소 (영상+이벤트) | 기본 대시보드 | 풀 관제 UI |
| DB | PostgreSQL | PostgreSQL | PostgreSQL / Tibero |
| 배포 | docker-compose | docker-compose | docker-compose / K8s |

> **언어 전환 핵심**: Phase 2에서 Java 백엔드 + DeepStream 추론 동시 도입. Phase 1의 Python 코드는 PoC 검증 후 참조 구현(reference implementation)으로 보존.

## 4. 기술 스택

### 4.1 선정 기준

- **한국 관급 표준 호환**: 조달청 나라장터 RFP, 전자정부 표준프레임워크(eGovFrame), GS인증 요건 충족
- **SL 개발팀이 실현 가능한 규모**: 1-2명으로 시작, 점진 확장
- **Phase별 언어 전환**: PoC는 Python으로 빠르게 검증 → 제품화 시 C++/Java로 전환 (업계 표준 패턴)
- 폐쇄망 온프레미스 배포
- 컨테이너 기반 모듈화

### 4.2 한국 관급 기술 표준 현실

한국 관급 관제시스템(도로공사, 국토부, 지자체)의 표준 기술 스택을 반영한다.

| 계층 | 한국 관급 표준 | 비고 |
|------|--------------|------|
| 영상 디코딩/추론 | **C++ (TensorRT, DeepStream)** | 인콘, 하이트론, KT AI Traffic 모두 C++ 추론 |
| 백엔드 API/관리 | **Java (Spring Boot / eGovFrame)** | 조달청 RFP에 "전자정부 표준프레임워크 적용" 빈발 |
| 프론트엔드 | **JavaScript (React/Vue)** | 관급 웹 표준 |
| DB | **Oracle / Tibero / PostgreSQL** | 국산 DBMS(Tibero) 우대. PG도 가능 |
| AI 모델 학습 | **Python (PyTorch)** | 학습은 Python이 글로벌/국내 표준 |

> **핵심**: Python은 AI 학습/프로토타이핑에 적합하나, 관급 납품 제품의 실시간 추론과 백엔드에는 **C++과 Java가 사실상 필수**.

### 4.3 Phase별 언어 전환 전략

```
Phase 0-1 (PoC, 카메라 1~3대)
  └─ 전체 Python — 빠른 검증이 목적
     추론: YOLO11 via ultralytics (Python/PyTorch)
     API: FastAPI (Python)
     ※ 업계 표준 패턴: KT, SK C&C도 프로토타입은 Python

Phase 2 (MVP, 카메라 10~20대)
  └─ 추론: NVIDIA DeepStream Python 바인딩 또는 ONNX/TensorRT
     백엔드: Java Spring Boot 도입 시작 (eGovFrame 호환)
     Python: 모델 학습/오프라인 분석 전용으로 역할 축소
     ★ DeepStream 마이그레이션 + Java 백엔드 구축 병행

Phase 3 (제품화, 카메라 50~100대+)
  └─ 추론: C++ DeepStream/TensorRT (필수 — Python으로 100대 불가)
     백엔드: Java Spring Boot (eGovFrame 기반)
     프론트: React + TypeScript
     Python: 모델 학습/재학습 파이프라인 전용
```

**Python 단독 처리 한계 (수치 기반)**:

| 구성 | 처리 가능 카메라 (15fps) |
|------|------------------------|
| Pure Python + OpenCV | **10~20대** |
| Python + DeepStream 바인딩 | **30~50대** |
| C++ + DeepStream/TensorRT | **100대+** |

> GIL(Global Interpreter Lock)로 인해 Python 단일 프로세스의 진정한 CPU 병렬 처리 불가. YOLO11 → ONNX → TensorRT 변환 경로가 이미 확립되어 있어 전환 비용은 예측 가능.

### 4.4 스택 상세 (Phase별)

| 레이어 | Phase 0-1 (PoC) | Phase 2 (MVP) | Phase 3 (제품) |
|--------|-----------------|---------------|----------------|
| **추론 엔진** | Python (ultralytics) | DeepStream/TensorRT | C++ DeepStream/TensorRT |
| **추론 언어** | Python | Python 바인딩 → C++ 전환 | **C++** |
| **백엔드 API** | FastAPI (Python) | **Java Spring Boot** 도입 | **Java Spring Boot (eGovFrame)** |
| **gRPC 서버** | grpcio (Python) | **Java gRPC** | **Java gRPC** |
| **프론트엔드** | React + TypeScript | React + TypeScript | React + TypeScript |
| **객체 감지** | YOLO11 (PyTorch) | YOLO11 (ONNX/TensorRT) | YOLO11 (TensorRT) |
| **객체 추적** | ByteTrack | ByteTrack | ByteTrack + BoT-SORT |
| **영상 디코딩** | FFmpeg (Python) | GStreamer/DeepStream | **GStreamer/DeepStream (C++)** |
| **DB** | PostgreSQL | PostgreSQL | PostgreSQL / Tibero |
| **시계열 (Phase 3)** | — | — | TimescaleDB (PG 확장) |
| **모델 서빙** | 직접 로드 | NVIDIA Triton | NVIDIA Triton |
| **컨테이너** | Docker + compose | Docker + compose | Docker + compose / K8s |
| **GPU (개발)** | NVIDIA RTX 4090+ | 동일 | 동일 |
| **GPU (센터)** | — | NVIDIA A100/L40S | NVIDIA A100/L40S |
| **엣지 (Phase 3+)** | — | — | NVIDIA Jetson Orin |

> **Phase 0~1에서 필요한 기술**: Python, PyTorch, YOLO11, ByteTrack, FFmpeg, FastAPI, React, Docker. 이것만으로 PoC 완성 가능. Phase 2에서 Java/C++ 전환 시작.

## 5. AI 모델 상세

### 5.1 객체 감지 (Detection)

- **모델**: YOLO11 (Ultralytics)
  - 센터: `yolo11l` (large) — 정밀도 우선
  - 엣지: `yolo11s` (small) — 속도 우선 (Phase 3+)
- **대상 클래스**: car, truck, bus, person, motorcycle, bicycle (COCO pretrained)
- **도메인 적응**: COCO pretrained → ITS CCTV 데이터 fine-tuning (Phase 2)
- **소형 객체 대응**: SAHI (Slicing Aided Hyper Inference)
  - 고속도로 카메라는 원거리 촬영으로 차량이 수십 픽셀 수준
  - **⚠️ SAHI는 추론 4~6배 증가 → Phase 0 벤치마크 후 도입 결정**
- **배포 포맷**: PyTorch → ONNX → TensorRT (점진 최적화)

```python
from ultralytics import YOLO

model = YOLO("yolo11l.pt")
model.train(data="its_highway.yaml", epochs=100, imgsz=1280)
model.export(format="onnx")
```

### 5.2 객체 추적 (Tracking)

- **기본**: ByteTrack — 경량, 고속도로 일정한 흐름에 적합
- **출력**: 각 트랙에 대해 track_id, bounding_box, velocity_vector, class

```python
results = model.track(frame, tracker="bytetrack.yaml", persist=True)
for box in results[0].boxes:
    track_id = box.id
    cls = box.cls
    xyxy = box.xyxy
```

### 5.3 역주행 판별 (F1 + F2)

모델이 아닌 **트래킹 후처리** 알고리즘:

1. 각 트랙의 이동 벡터 (최근 N프레임 위치 변화) 계산
2. "정상 방향" 결정:
   - **자동**: 초기 수 분간 트랙 방향 히스토그램에서 dominant direction 추출 → **양방향 통행 인식(F1)**
   - **노드링크 기반**: 전국표준노드링크에서 CCTV가 바라보는 도로 방향 참조
3. 정상 방향과 반대(±150° 이상)로 이동하는 트랙 → **역주행 판정(F2)**
4. 최소 지속 시간(2초 이상) 유지 시 이벤트 발생 (순간적 방향 변화 필터링)

### 5.4 보행자/이륜차 감지 (F3)

- YOLO11의 COCO 클래스 중 `person`, `motorcycle`, `bicycle` 검출
- **고속도로에서 검출 = 즉시 이벤트** (고속도로에 보행자/자전거가 있으면 그 자체가 이상)
- 오탐 대응: 갓길/휴게소 영역 마스킹 (카메라별 설정)

### 5.5 정체/사고 감지 (F4)

**정체 판별 (룰 + VDS 퓨전)**:
- 트래킹된 차량 평균 속도 < 임계값 (예: 20km/h) × N초 이상 지속
- 단위 면적당 검출 차량 수(밀도) 증가
- VDS 소통정보 API 속도 데이터와 교차 확인

**사고 감지 (룰 기반)**:
- 차량 급정지 (속도 급감 + 정지 지속)
- 비정상 위치 정차 (갓길이 아닌 주행 차로)
- 주변 차량의 회피 기동 패턴
- 공사정보 API와 교차 → 공사 구간이면 오탐 필터링

### 5.6 악천후 대응 (F5)

악천후에서도 F1~F4가 동작하도록 감지 파라미터를 동적 조정한다.

**1차: 기상청 API 활용 (비용 0, 구현 간단)**
- CCTV 좌표 기반 현재 날씨 조건 자동 파악
- 안개/비/눈 조건에 따라 감지 임계값 자동 조정

**2차: 영상 기반 날씨 분류 (Phase 2+)**
- **모델**: MobileNetV3 또는 EfficientNet-B0 (경량 CNN)
- **클래스**: 맑음 / 비 / 눈 / 안개 / 야간 계열
- **실행 주기**: 30초~1분 간격
- 기상청 API와 교차 확인 (국지적 안개 등 보완)

```
날씨 조건별 파라미터 조정 방향:
- 안개/야간 → detection confidence threshold 하향 (recall 우선)
- 비/눈     → NMS 강화 (물방울/눈발 오검출 방지)
- 전체      → 이벤트 알림에 환경 컨텍스트 자동 포함
```

> **구체적 threshold 수치는 Phase 2에서 실험을 통해 결정한다.**

### 5.7 모델 파이프라인 요약

```
[CCTV 영상 프레임]
      │
      ▼
[날씨 조건 확인] ──→ 파라미터 동적 조정
(기상청 API, Phase 2+: 영상 분류 보조)
      │
      ▼
[SAHI 타일링] (Phase 2+, 벤치마크 통과 시)
      │
      ▼
[YOLO11 Detection]
      │
      ▼
[ByteTrack Tracking]
      │
      ├──→ [F1: 양방향 통행 인식] ──→ 카메라별 정상 방향 학습
      ├──→ [F2: 역주행 판별 룰] ──→ 이벤트
      ├──→ [F3: 보행자/이륜차] ──→ 이벤트 (고속도로에 있으면 즉시)
      ├──→ [F4: 정체 판별 + VDS 퓨전] ──→ 이벤트
```

## 6. 데이터 퓨전

영상 분석 단독보다 공개 API 데이터를 결합하면 정확도와 신뢰도가 크게 향상된다.

### 6.1 퓨전 시나리오

| 시나리오 | 영상 분석 | 보조 데이터 | 퓨전 결과 |
|----------|-----------|-------------|-----------|
| 정체 확인 | 차량 속도 감소 감지 | VDS 속도 급감 | 정체 확신도 ↑ |
| 미감지 사고 | 차량 정지 감지 | 돌발정보 API에 해당 구간 사고 없음 | 미감지 사고 → 긴급 알림 |
| 안개 대응 | 영상 선명도 하락 | 기상청 안개 예보 | 감지 임계값 자동 조정 |
| 공사 오탐 필터 | 차량 정지/저속 | 공사정보 API 해당 구간 공사 중 | 오탐 제거 |

### 6.2 CCTV 캘리브레이션 자동화

카메라별 "정상 주행 방향" 자동 설정:

1. CCTV 좌표 → 전국표준노드링크에서 가장 가까운 링크 매칭
2. 해당 링크의 방향 정보 (시점노드 → 종점노드) 참조
3. 초기 수 분간 트랙 방향 히스토그램과 교차 검증
4. 불일치 시 관리자 확인 요청

## 7. 이벤트 관리

### 7.1 이벤트 신뢰도 등급

| 등급 | 조건 | 관제사 액션 |
|------|------|-------------|
| 긴급 (CRITICAL) | conf ≥ 0.85 + 다중 소스 확인 | 즉시 팝업 + 사운드 알림 |
| 주의 (WARNING) | conf 0.60~0.85 또는 단일 소스 | 이벤트 목록 상단 표시 |
| 확인 필요 (INFO) | conf 0.40~0.60 | 이벤트 로그 기록, 수동 확인 가능 |

### 7.2 오탐 관리

오탐이 빈번하면 관제사가 알림을 무시하게 되고 시스템은 사장된다.

**원칙: Precision First**
- 초기 목표: Precision ≥ 90% (놓치는 것보다 잘못 알리는 것이 더 치명적)
- 관제사 피드백 루프: 이벤트에 [정탐]/[오탐] 마킹 → 패턴 분석 → 파라미터 조정

**오탐 빈발 조건 사전 필터링**:

| 조건 | 오탐 유형 | 대응 |
|------|-----------|------|
| 공사 구간 | 정지 → 사고 오탐 | 공사정보 API 자동 필터링 |
| PTZ 카메라 회전 | 모든 객체 급이동 | 프레임 간 전체 이동량 감지 → 카메라 이동으로 판단 |
| 야간 헤드라이트 | 반사 → 오검출 | 최소 프레임 지속 조건 |
| 비/눈 | 물방울/눈발 → 오검출 | 기상 조건에 따른 threshold 상향 |

**알림 피로도 관리**:
- 중복 이벤트 병합: 동일 카메라 + 동일 유형 30초 내 → 1건
- 카메라당 분당 최대 알림: 5건 (설정 가능)

### 7.3 감사 추적 (Audit Trail)

관급 납품 필수 요구사항:

- 이벤트 발생 전후 10초 영상 클립 자동 저장
- 감지 로그: 어떤 모델이, 어떤 프레임에서, 어떤 confidence로 판단했는지
- 데이터 퓨전 근거: 어떤 보조 데이터가 판단에 영향을 미쳤는지
- 관제사 조치 기록: 확인/조치/무시 시간 및 담당자
- 보존 기간: 최소 1년 (설정 가능)

## 11. 기술적 주의사항

### 11.1 ITS CCTV 영상 품질

- 해상도/프레임레이트가 카메라별로 상이. 일부는 매우 낮을 수 있음
- HLS 스트림 URL은 API 호출 시마다 변경될 수 있음 → 주기적 갱신 필요
- API 호출 횟수 제한이 있을 수 있음 → 캐싱 전략 필요

### 11.2 실시간 성능

- Detection + Tracking: ≥ 15 FPS (카메라당, SAHI 미적용)
- SAHI 적용 시: ≥ 8 FPS (타일 수에 따라 변동)
- 역주행 판별: ≤ 2초 지연
- 사고 감지 → 알림: ≤ 5초

### 11.3 개인정보보호

- 번호판/얼굴: 학습 데이터 내 비식별화 처리 필수
- 이벤트 클립 저장 시 개인정보 마스킹 옵션
- Phase 0에서 법률 검토 착수 (개인정보보호법, 영상정보처리기기 가이드라인)

### 11.4 보안 (관급 납품 시)

- 폐쇄망 배포: AI 서버는 관제망 내부, 외부 인터넷 차단
- 모델 업데이트: 오프라인 업데이트 절차
- 접근 통제: 역할별 권한 분리
- GS인증/보안심사: 파트너사 주도, SL 기술 지원

### 11.5 관급 표준 준수 (Korean Government Standards)

관급 납품 목표이므로 아래 한국 정부 IT 표준을 Phase별로 충족해야 한다.

**전자정부 표준프레임워크 (eGovFrame)**:
- 조달청 나라장터 RFP에서 "전자정부 표준프레임워크 적용" 빈발
- Java Spring 기반. Phase 2에서 Java 백엔드 도입 시 eGovFrame 호환 구조로 설계
- eGovFrame 공식 사이트: https://www.egovframe.go.kr/

**GS인증 (Good Software)**:
- 기능성, 신뢰성, 사용성, 효율성, 유지보수성, 이식성, 보안성 7개 항목 평가
- 언어 제한 없으나, Java/C++ 프로젝트의 인증 전례가 압도적
- Phase 3에서 인증 준비 (파트너사 주도)
- 소스코드 정적 분석 도구: Java — SonarQube/PMD, C++ — Coverity/cppcheck

**CC인증 (Common Criteria)**:
- 보안 기능이 포함된 제품에 필요할 수 있음 (접근 통제, 감사 추적 등)
- 국정원 가이드라인 준수 (암호 모듈, 접근 통제)

**DB 호환성**:
- 공공기관은 Oracle 또는 국산 DBMS(TmaxSoft Tibero) 사용이 일반적
- PostgreSQL도 허용되나, Tibero 호환성 확보 시 경쟁력 상승
- JPA/Hibernate 사용 시 DB 교체가 용이 → Phase 2 Java 전환 시 반영

**Phase별 준수 로드맵**:

| 표준 | Phase 0-1 | Phase 2 | Phase 3 |
|------|-----------|---------|---------|
| eGovFrame | N/A (Python PoC) | Java 백엔드 eGovFrame 호환 설계 | eGovFrame 적용 |
| GS인증 | N/A | 코드 품질 관리 시작 | 인증 신청 |
| CC인증 | N/A | 보안 요건 분석 | 대응 (필요 시) |
| DB 호환 | PostgreSQL | PostgreSQL + JPA | + Tibero 검증 |
| 망분리 | 개발망 | 폐쇄망 배포 테스트 | 폐쇄망 완전 동작 |

## 12. Claude Code 작업 지침

이 프로젝트 작업 시 아래 원칙을 따르세요:

1. **한국어 주석/문서**: 코드 주석, 커밋 메시지, 문서는 한국어로 작성
2. **엔진/앱 분리**: `engine/`은 독립 모듈. `app/`은 독립 제품 전용. 혼합하지 않음
3. **설정 외부화**: 하드코딩 금지. 환경변수 또는 config.yaml로 관리
4. **API 키 보안**: ITS API 키, DB 접속정보 등은 절대 코드에 포함하지 않음
5. **Docker 우선**: 모든 서비스는 Dockerfile 포함. docker-compose로 로컬 개발환경
6. **테스트 데이터**: 실제 API 키 없이도 샘플 영상으로 테스트 가능하도록 mock 제공
7. **Phase 0부터 시작**: 전체를 한 번에 만들지 않음
8. **Precision First**: 새 기능 추가 시 오탐 증가 여부를 항상 확인
9. **gRPC proto 유지**: `engine/api/proto/highlink.proto`를 인터페이스의 Single Source of Truth로 유지
10. **관급 표준 인식**: Phase 0-1은 Python PoC이지만, 제품화 시 C++(추론)+Java(백엔드) 전환이 예정됨. Python 코드 작성 시에도 **인터페이스와 데이터 모델을 깔끔하게 유지**하여 Java/C++ 전환 시 참조 구현으로 활용 가능하도록 할 것
11. **eGovFrame 대비**: Phase 2 Java 전환 시 전자정부 표준프레임워크 호환 구조로 설계. Spring Boot + JPA + REST/gRPC 표준 패턴 준수
