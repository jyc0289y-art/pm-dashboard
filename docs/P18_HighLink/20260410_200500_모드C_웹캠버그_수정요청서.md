## 1. 버그 현상

**모드 C (실시간 CCTV 감지)**에서 ITS API로 받아온 CCTV 스트림 URL을 사용하지 않고, **클라이언트(브라우저)의 웹캠을 활성화**하여 사용자 본인을 보행자로 감지하는 현상 발생.

## 2. 원인 분석

`app/streamlit/app.py`의 모드 C 구현부에서 `streamlit-webrtc`의 `webrtc_streamer()`를 사용하고 있음:

```python
webrtc_streamer(
    key="highlink-cctv",
    video_processor_factory=HighLinkVideoProcessor,
    media_stream_constraints={"video": True, "audio": False},
)
```

**`webrtc_streamer`는 WebRTC 프로토콜 기반으로 동작하며, `media_stream_constraints={"video": True}`는 브라우저의 `getUserMedia()` API를 호출하여 클라이언트 측 카메라(웹캠)를 캡처합니다.** 서버 측에서 ITS CCTV URL을 `cv2.VideoCapture()`로 읽는 것과는 완전히 다른 경로입니다.

즉, CCTV 드롭다운에서 URL을 선택해도 실제 영상 소스는 **사용자의 노트북 웹캠**이 되는 구조적 버그입니다.

## 3. 수정 방향

`webrtc_streamer()` 블록을 **완전히 제거**하고, 서버 측에서 직접 CCTV URL에 연결하는 방식으로 변경해야 합니다:

```python
# 변경 전 (버그): 브라우저 웹캠 캡처
webrtc_streamer(
    key="highlink-cctv",
    video_processor_factory=HighLinkVideoProcessor,
    media_stream_constraints={"video": True, "audio": False},
)

# 변경 후 (수정): 서버 측 CCTV URL 직접 연결
if st.button("▶️ 감지 시작", type="primary"):
    model, device = load_model()
    cap = cv2.VideoCapture(selected_cctv.url)
    stframe = st.empty()

        # YOLO 추론 + reclassify_truck + 오버레이

```

### 핵심 포인트:
- `streamlit-webrtc` 의존성 자체를 모드 C에서 제거
- `cv2.VideoCapture(selected_cctv.url)`로 ITS CCTV 스트림 직접 수신
- 서버 측 추론 → `st.image()`로 결과 프레임 렌더링
- `reclassify_truck()` 및 `load_classifier()` 2단계 분류 로직은 이 루프 안에 통합

## 4. 참고사항

- 이 버그는 Windows PC에서 데모 시연 중 발견됨
- `streamlit-webrtc`는 모드 A(웹캠 테스트)에서는 정상 사용 가능하나, 모드 C(CCTV 연결)에서는 용도에 맞지 않음
- 최근 추가된 `load_classifier()` 2단계 분류기도 수정된 OpenCV 루프 안에 통합 필요
