## 총평 — 매뉴얼의 인터랙티브 교육 도구 진화와 RR 이중보정 버그 발견

### 이번 세션의 의의

P10.8wc는 두 차원에서 중요하다. 첫째, P10.5wc 매뉴얼을 **13개 인터랙티브 요소**(SVG 다이어그램, 계산기, 시뮬레이터, 애니메이션)로 강화하여 참조 문서를 **교육 도구**로 격상시켰다. base64 임베드로 3.8MB 단일 HTML 완성. LMI 운영 가이드(허용/주의/금지 색상 분류), Background Update 절차, 크로스토크 업데이트, Normal QC 선원 도입 가이드까지 추가하여 직원이 독립적으로 장비를 운용할 수 있는 수준의 문서를 완성했다.

둘째, **RR 이중보정 버그**를 실측 데이터(Raw α=9/β=101 vs RR α=6/β=27)로 발견하고 3가지 수정 옵션까지 제시한 것은 P10.14wc의 완벽한 사전 준비였다.

## 포크 가이드 — Hodlum RR 파싱 버그 수정

**이 포크는 P10 프로젝트 내 다음 순번(P10.14wc)으로 부여할 것.**

### 문제 정의
Hodlum이 RR 응답의 α/β 값을 항상 Raw 스케일러 카운트(정수 펄스 수)로 해석하지만, 장비가 Bq 모드일 때 RR 응답은 **Bq 표시값 × 100** (소수점 제거 정수)을 반환함. 이로 인해 이미 장비가 BG/XT/효율 보정을 완료한 최종값에 대해 Hodlum이 다시 동일 보정을 적용하는 **이중 보정** 발생.

### 검증 데이터 (2026-03-19 실측)
```
실제 Raw (Scaler 모드 확인): α=9, β=101 (2분 측정)
장비 BG: α=3 cpm, β=45 cpm (QC BG Update로 갱신 완료)
장비 Bq: α=0.06, β=0.28 (검산 일치 — 정확)

RR 응답 (Bq 모드): α=000006, β=000027  ← 0.06×100, 0.27×100
Hodlum 해석: Raw=6, Raw=27 ← 틀림! 실제 Raw는 9, 101
Hodlum 결과: 이중 보정된 잘못된 Bq 값
```

### RR 응답의 모드별 α/β 값 해석 (제작사 매뉴얼 + 실측 확인)
| 모드 | RR의 x 필드 | α/β 값 의미 | 소수점 처리 |
|------|-----------|-----------|-----------|
| Scaler | S | 실제 펄스 수 (정수) | 없음 |
| CPM | C | CPM × 10 | ÷10 |
| Bq | B (또는 D) | Bq × 100 | ÷100 |
| CPS | P | CPS × 100 | ÷100 |
| DPM | D | DPM (정수) | 없음 |

**주의: 3030-2에서 Bq 모드여도 RR의 x 필드가 "D"로 반환될 수 있음** (매뉴얼 7.5절 참조)

### 수정 방향 (3가지 옵션)

**옵션 A: 측정 전 Scaler 전환 (권장)**
1. 측정 전 `D0` 전송 (Scaler 모드 전환)
2. C4/C5 측정 → RR 수신 (항상 Raw 펄스)
3. 측정 후 `D3` 전송 (원래 모드 복귀)
- 장점: Hodlum의 기존 계산 로직(Raw→전체 8메트릭) 그대로 유지
- 단점: SET 명령(D0/D3) 사용 → "READ-only" 원칙 약간 위배 (교정값 변경 아님, 표시 모드만 변경이므로 안전)

**옵션 B: RR 모드 필드 기반 역산**
1. RR의 x 필드 확인 (S/C/B/P/D)
2. 모드에 따라 α/β 값 해석 변경
3. Bq 모드면: Bq = value/100 → 역산으로 Raw 추출 → 8메트릭 재계산
- 장점: SET 명령 불필요
- 단점: 역산 시 반올림 오차 누적 가능. D 필드의 모호성(DPM vs Bq)

**옵션 C: 장비 크롤링 방식 (원래 취지)**
1. RR의 x 필드로 모드 확인
2. 해당 모드의 값을 그대로 표시 (Bq 모드면 Bq만 표시)
3. 나머지 메트릭은 파라미터(BG/Eff/XT)를 사용해 역산
- 장점: 장비 계산값을 신뢰, 이중 보정 원천 차단
- 단점: 8메트릭 중 일부는 역산 (정확도 약간 저하)
