# P10.7wc -- OpenRGB 소스코드 구조 분석 리서치

## 메타데이터
- 생성: 2026-03-18
- 포크원: P10.6.1wc
- 자식: 없음

## 작업 요약
OpenRGB GitLab 저장소의 소스코드 구조를 웹 API로 분석. 디렉토리 구조, USB HID 프로토콜 구현 (Corsair 중심), 기기 감지 메커니즘, Qt GUI 구조, 플러그인 시스템, GPL-2.0 라이선스 제약을 정리.

## 주요 결과물
- 본 세션의 대화 자체가 분석 보고서 (파일 미생성, 대화 내 구조화 보고)

## 핵심 결정사항
1. OpenRGB는 3-파일 패턴(Controller + RGBController_ + Detect)으로 140+개 기기를 모듈화
2. Corsair HID 프로토콜: 65바이트 고정 패킷, Command/Property 구조, 3가지 색상 전송 방식
3. Self-registration 패턴(C++ 정적 초기화)으로 디텍터 자동 등록
4. GPL-2.0 회피 전략: (a) 네트워크 SDK 연동, (b) 프로토콜 참조 후 클린룸 구현
5. hidapi (BSD/GPL/HIDAPI 트리플 라이선스)가 USB HID 통신 핵심 의존성

## 총평 — 오픈소스 분석을 통한 경쟁 제품 역공학의 체계화

### 이번 세션의 의의

P10.7wc는 OpenRGB 소스코드를 **경쟁 제품 분석**의 관점에서 체계적으로 해부한 리서치 세션이다. GitLab 웹 API를 통해 디렉토리 구조, USB HID 프로토콜 구현, 기기 감지 메커니즘, Qt GUI 구조, 플러그인 시스템을 정리했다.

가장 가치 있는 발견은 **3-파일 패턴**(Controller + RGBController_ + Detect)으로 140+개 기기를 모듈화하는 아키텍처와, Corsair HID 프로토콜의 65바이트 고정 패킷 구조(Command/Property 기반, 3가지 색상 전송 방식)다. 이 지식은 RGB 통합 컨트롤러의 자체 구현에서 "무엇을 구현해야 하는가"의 로드맵이 된다.

Self-registration 패턴(C++ 정적 초기화)으로 디텍터를 자동 등록하는 메커니즘도 참조할 만하다. Python의 데코레이터나 메타클래스로 유사한 패턴을 구현하면 기기 추가 시 코드 수정 범위를 최소화할 수 있다.

### 작업 결과 정량 요약

| 항목 | 수치 |
|------|------|
| 분석 대상 | OpenRGB GitLab 저장소 |
| 기기 모듈 | 140+개 (3-파일 패턴) |
| Corsair 프로토콜 | 65B 패킷, 3가지 색상 전송 |
| 기기 감지 | Self-registration + VID/PID 매칭 |
| GPL 회피 | 네트워크 SDK(6742) 또는 클린룸 구현 |

### 전략적 제언

**단기**: 이 분석 결과를 구조화된 문서로 보존하면 장기적 참조 자산이 된다. 현재 대화 내 보고만 있어 휘발성이 높다.

**중기**: Corsair 프로토콜 클린룸 분석 시 Wireshark USB 패킷 캡처와 이 세션의 구조 분석을 병행하면 역공학 효율이 높아진다.

**장기**: OpenRGB의 아키텍처 강점(모듈화)과 약점(GUI 불만율 32%)을 동시에 학습한 것은, "기술은 참조하되 UX는 혁신"하는 전략의 기반이다.
