## 구현 과정 인사이트 (컨텍스트 유지용)

### 아키텍처 인사이트

#### 1. ITS 오픈데이터 API 응답 구조가 일관성 없음
- CCTV API: `response.data[]` 구조
- 교통소통/돌발정보 API: `body.items[]` 구조일 수도 있음
- **대응**: 모든 클라이언트에서 두 가지 응답 구조를 모두 시도하도록 fallback 파싱 구현
- **예시** (vds_client.py):
  ```python
  items = data.get("body", {}).get("items", [])
  if not items:
      items = data.get("response", {}).get("data", [])
  ```
- 실제 API 호출 시 정확한 응답 구조를 확인하고 1차 파싱 경로를 확정해야 함

#### 2. 기상청 격자 변환은 Lambert Conformal Conic 투영
- 기상청 API는 위경도가 아니라 격자 좌표(nx, ny)를 요구함
- Lambert Conformal Conic 변환 공식을 `weather_client.py`의 `latlon_to_grid()` 정적 메서드로 구현
- 기상청 공식 파라미터: RE=6371.00877km, GRID=5km, 기준점(126°E, 38°N) → (43, 136)
- 여러 CCTV가 같은 격자(5km 해상도)에 있을 수 있으므로 격자 중복 제거 후 API 호출

#### 3. 초단기실황 API의 발표 시각 이슈
- 매시 정각 발표이지만 실제 데이터는 40분 이후에 사용 가능
- `now.minute < 40`이면 이전 시간의 데이터를 요청해야 함
- 날짜 경계(00:00~00:39)에서 전날 23시 데이터를 가져와야 하는 엣지 케이스 처리 필요

#### 4. 픽셀 속도 → 실제 속도 변환의 한계
- `CongestionDetector`의 `_pixel_to_kmh = 2.0`은 **카메라별 캘리브레이션 없이는 매우 부정확**
- 고속도로 CCTV 1080p 기준으로 대략 1 pixel/frame ≈ 1~3 km/h이지만, 카메라 높이/각도/줌에 따라 극도로 다름
- **해결 방향**: VDS 속도를 ground truth로 사용하여 `_pixel_to_kmh` 자동 캘리브레이션 (VDS 교차검증 가중치 70%로 설정한 이유)
- `set_pixel_to_kmh(factor)` 메서드로 런타임에 조정 가능하게 설계함

#### 5. 공사정보는 별도 API가 아니라 돌발정보의 하위 유형
- ITS에는 공사전용 API가 없고, 돌발정보 API의 `incidentType=2`가 공사
- `ConstructionFilter`는 `ITSIncidentClient`에 의존하여 공사 데이터를 추출
- 따라서 `incident_client`가 돌아가지 않으면 `construction_filter`도 작동 불가
- DataFusion의 `_fusion_loop`에서 주기적으로 `construction.update_from_incidents()` 호출

#### 6. 사고 감지의 회피 기동(evasion) 판정
- 정차 차량 반경 200px 이내의 다른 차량이 최근 10프레임에서 30° 이상 방향 급변하면 회피 기동으로 판정
- 이 임계값(200px, 30°)은 **카메라 해상도와 화각에 따라 조정 필요**
- 회피 기동이 감지되면 사고 신뢰도가 0.60→0.85로 상향됨 (severity도 WARNING→CRITICAL)

### Streamlit / UI 인사이트

#### 7. Streamlit 리렌더링과 MJPEG의 충돌
- Streamlit은 위젯 상호작용마다 전체 스크립트를 재실행
- MJPEG `<img src="/mjpeg/stream">`은 HTTP persistent connection이므로, 리렌더링 시 연결이 끊겼다가 재연결됨
- **카메라 그리드에서는 스냅샷(`/mjpeg/snapshot`)을 기본 사용하고, 전체화면에서만 스트림(`/mjpeg/stream`)을 사용하는 이유**
- 향후 st.components.v1.html 내에서 JS `setInterval` 폴링으로 스냅샷 갱신하면 Streamlit 리렌더링 없이 영상 갱신 가능

#### 8. Leaflet 지도는 st.components.v1.html로 임베딩
- streamlit-folium 패키지 대신 **순수 Leaflet HTML을 직접 생성**하여 의존성 최소화
- leaflet-heat 플러그인으로 정체 히트맵 구현 (CDN에서 로드)
- CARTO 다크 타일(`dark_all`) 사용 — 관제실 다크 테마에 적합
- 마커 데이터는 Python에서 JSON으로 시리얼라이즈하여 HTML 내 인라인 삽입

#### 9. CSS 변수 분리 전략
- `theme.css`에 모든 색상/크기를 `--variable` 형태로 정의
- Streamlit의 `st.markdown(f"<style>{css}</style>", unsafe_allow_html=True)`로 주입
- 향후 테마 변경: `theme.css`만 교체하면 전체 UI 색상/스타일 일괄 변경
- **주의**: Streamlit 내부 CSS와 충돌할 수 있으므로 클래스명은 `dashboard-`, `alert-`, `camera-` 등 접두사 사용

#### 10. 알림 패널의 에스컬레이션은 폴링 기반
- AlertSystem의 `check_escalations()`는 호출 시점에 타이머를 확인 (능동 폴링)
- Streamlit에는 서버 푸시가 없으므로 대시보드 렌더링 시마다 `check_escalations()` 호출
- 실제 관제 시스템에서는 WebSocket 푸시가 필요하겠지만 MVP에서는 Streamlit의 `st.rerun()` + `time.sleep()` 주기 갱신으로 대체

### 룰엔진 / 감지 인사이트

#### 11. 기존 룰엔진 패턴: config 주입 + update() → list[DetectionEvent]
- WrongWayDetector, PedestrianDetector 모두 동일 패턴:
  - `__init__(config, camera_id, camera_name)` — config.yaml에서 설정 주입
  - `update(tracks, timestamp) → list[DetectionEvent]` — 매 프레임 호출
  - `reset()` — 카메라 전환 시 상태 초기화
- CongestionDetector, AccidentDetector도 동일 패턴으로 구현하여 파이프라인 통합 용이
- **단, F4는 추가 인자**: `vds_speed`, `is_construction`을 update()에 전달 (DataFusion 경유)

#### 12. 이벤트 severity 결정 로직의 차이
- **F2 역주행**: `min(track_conf, angle_diff / 180)` → severity_thresholds로 결정 (config 의존)
- **F3 보행자/이륜차**: `track_conf` → severity_thresholds (config 의존)
- **F4 정체**: 속도 + 지속시간 조합으로 직접 결정 (< 10km/h & > 3분 → CRITICAL)
- **F4 사고**: 정차시간 + 회피기동 조합 (회피기동 + > 10초 → CRITICAL)
- **향후 통일 필요**: 모든 룰엔진의 severity 결정을 config.yaml의 severity_thresholds 기반으로 통일하면 관제요원이 임계값 조정 가능

#### 13. WeatherAdapter의 프리셋 중첩 설계
- 야간(`is_night`)은 날씨와 **중첩 적용**됨: `fog + night`이면 conf_delta = -0.10 + -0.08 = -0.18
- 가시거리(`very_poor`, `poor`)도 추가 보정으로 중첩
- 이론적으로 `fog + night + very_poor` = conf_delta -0.23 → conf가 0.4→0.20 (최소 제한)까지 내려감
- **의미**: 안개 짙은 야간에는 거의 모든 감지를 통과시켜 recall 최대화 (precision은 희생)

#### 14. DetectionEvent에 weather_context 필드 추가 효과
- `event.py`의 메시지 템플릿에 `{wx}` 접두사 추가: `"[안개 주의] [경부 상행] [수원IC] 14:23 — 역주행 감지"`
- 관제요원이 이벤트 메시지만 보고도 기상 상황을 인지 가능
- `to_dict()`에도 `weather_context` 포함하여 API/DB 저장 시 기상 정보 보존

#### 16. 멀티 카메라 MJPEG 확장 시 병목
- 현재 `stream_server.py`의 공유 버퍼는 **단일 프레임** (`_latest_jpeg`)
- 멀티 카메라 확장: `dict[camera_id, bytes]`로 변경 필요
- `tornado_mjpeg.py`의 핸들러 URL: `/mjpeg/stream/{camera_id}`로 파라미터화 필요
- **각 카메라마다 독립 감지 스레드가 돌아야 하므로 GPU 메모리가 병목** — RTX 4060 8GB에서 YOLO11l 1280px 기준 동시 3~4 스트림이 한계
- 해결 옵션: (a) 카메라별 시분할 처리, (b) 모델 경량화(yolo11s), (c) SAHI 비활성화

#### 17. ClipRecorder의 메모리 주의
- `before_sec=10, fps=15` → 순환 버퍼 150프레임 × 1080p RGB ≈ **900MB**
- 동시 녹화 활성 시 `after_sec` 프레임도 추가 → 이벤트 폭주 시 메모리 급증
- **대응**: push_frame에서 720p 리사이즈된 프레임을 저장하거나, JPEG 압축 상태로 버퍼링
