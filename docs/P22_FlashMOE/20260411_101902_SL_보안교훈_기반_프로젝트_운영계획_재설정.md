# Flash-MoE 397B on 32GB M5 MacBook Pro — 최적화 전략 핸드오프 문서

## 프로젝트 목표

**Qwen3.5-397B-A17B (397B 파라미터 MoE 모델)를 MacBook Pro M5 (32GB / 1TB SSD)에서 최대 속도로 로컬 구동하기**

---

## 1. 하드웨어 스펙

| 항목 | 값 |
|---|---|
| 칩 | Apple M5 (base, not Pro/Max) |
| 통합 메모리 | 32GB |
| SSD | 1TB NVMe |
| SSD 읽기 속도 | ~6.3 GB/s (M4 대비 211% 향상) |
| SSD 쓰기 속도 | ~6.0 GB/s |
| macOS | 26.x |

### 핵심 하드웨어 제약

- **Apple Silicon 통합 메모리 버스**: SSD DMA와 GPU가 메모리 컨트롤러(~400 GB/s)를 공유함. 단순 prefetch는 GPU 레이턴시 스파이크를 유발하므로 **예측 기반 선택적 prefetch**만 효과적
- **페이지 캐시 가용량**: OS(~5GB) + 비전문가 가중치(~5.5GB) = ~11GB 상주 → 페이지 캐시 ~18-20GB 가용
- **원본 Flash-MoE 대비**: M3 Max 48GB 기준 페이지 캐시 ~37GB, SSD 17.5 GB/s

## 2. 모델 아키텍처 (Qwen3.5-397B-A17B)

| 항목 | 값 |
|---|---|
| 총 파라미터 | 397B |
| 토큰당 활성 파라미터 | 17B (원본) / Flash-MoE에서 K=4 사용 |
| 레이어 수 | 60 (45 GatedDeltaNet + 15 Full Attention) |
| 레이어당 전문가(expert) 수 | 512 |
| 활성 expert 수 | K=10 (원본) → K=4 (Flash-MoE) → K=3 (실험적) |
| Hidden dimension | 4096 |
| 컨텍스트 길이 | 262K (네이티브) |
| 비전문가 가중치 크기 | ~5.5GB (임베딩, 라우팅 매트릭스 등) |
| 4-bit 양자화 시 전체 크기 | ~209GB on disk |
| Q3 GGUF 양자화 시 | ~150GB on disk (추정) |
| 2-bit 양자화 시 | ~120GB on disk |

### MoE 스트리밍의 핵심 원리

토큰당 전체 파라미터의 ~4.3%만 활성화. 비활성 expert 가중치를 SSD에 두고, 필요한 K=4개 expert만 실시간 로드. 각 expert 크기: ~6.75MB (4-bit) / ~5MB (Q3).

## 3. 기존 구현체 분석

### 3.1 원본 Flash-MoE (danveloper/flash-moe)

- **GitHub**: https://github.com/danveloper/flash-moe
- **구현**: 순수 C / Objective-C / Metal (Python 없음, 프레임워크 없음)
- **성능**: M3 Max 48GB에서 4.4 tok/s (4-bit), 5.7 tok/s (2-bit)
- **RAM 사용량**: ~5.5-6GB 상주 (비전문가 가중치 mmap)
- **캐싱 전략**: "Trust the OS" — macOS 페이지 캐시에 전적으로 의존, 71% 히트율
- **파이프라인**: 엄격한 직렬 (GPU → SSD → GPU) — 병렬화 시 오히려 성능 저하
- **실패한 최적화 (58개 실험 기록)**:
  - LZ4 압축: 13% 성능 저하 (해제 오버헤드)
  - Custom Metal LRU 캐시: 38% 성능 저하 (OS 캐시와 경쟁)
  - Speculative expert prefetching: 73% 성능 저하 (GPU contention)
- **성공한 최적화**:
  - FMA-optimized dequant kernel: +12%
  - Accelerate BLAS for linear attention: +64% vs scalar
  - Deferred GPU compute: GPU 유휴 시간 활용

### 3.2 gorroai/flash-moe (Anemll 포크) ⭐ 핵심 레퍼런스

- **GitHub**: https://github.com/gorroai/flash-moe
- **성능**: M5 Max 128GB에서 **20.34 tok/s** (원본 대비 4.6배)
- **방법론**: Claude Code autoresearch loop — 43개 Metal 최적화 실험
- **핵심 혁신**:

#### Temporal Expert Prediction (+55% 속도)
- 토큰 간 라우팅 상관관계(27%)를 이용해 다음 레이어의 expert를 GPU 계산 중에 미리 SSD에서 prefetch
- 원본의 "직렬만 가능" 제약을 **예측 기반 선택적 병렬화**로 우회
- 단독 기여: 10.61 → 16.40 tok/s (Exp9)

#### Q3 GGUF Experts (+36% 속도)
- 4-bit 대비 36% 빠른 디코딩 (12.9 vs 9.5 tok/s)
- 품질 거의 동일: PPL 3.81 (Q3) vs 3.64 (4-bit) vs 3.49 (Q8)
- 파일 크기 감소 → 페이지 캐시 히트율 향상에 유리

#### cache-io-split 4
- SSD I/O를 4개 청크로 분할하여 읽기 효율 최적화

#### GGUF Embedding + LM Head
- embedding_q8_0.bin + lm_head_q6.bin 사용
- PPL 개선 효과, 추가 비용 거의 없음 (1.4ms/tok, 디코드 시간의 2%)

#### 실행 명령:
```bash
./metal_infer/infer \
  --model ~/Models/flash_mlx_4bit \
  --q3-experts \
  --predict \
  --cache-io-split 4 \
  --gguf-embedding ~/Models/flash_mlx_4bit/gguf/embedding_q8_0.bin \
  --gguf-lm-head ~/Models/flash_mlx_4bit/gguf/lm_head_q6.bin \
  --prompt "Hello" --tokens 100 --stream
```

#### RAM 사용량 테이블 (gorroai 기준, 48GB 기준):
| 컨텍스트 | KV 캐시 | 총 RAM | 페이지 캐시 여유 |
|---|---|---|---|
| 8K | 0.3GB | 6.1GB | 41.9GB |
| 128K | 4.0GB | 9.9GB | 38.1GB |
| 256K | 8.1GB | 13.9GB | 34.1GB |

**32GB에서 환산:**
| 컨텍스트 | KV 캐시 | 총 RAM | 페이지 캐시 여유 |
|---|---|---|---|
| 2K | ~0.1GB | ~5.9GB | ~26.1GB |
| 4K | ~0.2GB | ~6.0GB | ~26.0GB |
| 8K | 0.3GB | 6.1GB | ~25.9GB |

→ 컨텍스트 2K-8K 사용 시 페이지 캐시 ~26GB 확보 가능

### 3.3 FlashTier-MoE

- **GitHub**: https://github.com/w3t-wr3/FlashTier-MoE
- 3가지 양자화 설정을 ~10초 만에 핫스왑 (K3 2-bit, K3 4-bit, K4 4-bit)
- Open WebUI 연동
- 테스트/비교에 유용

### 3.4 omlx (Apple Silicon LLM 추론 서버)

- **GitHub**: https://github.com/jundot/omlx
- SSD 캐싱 (cold tier), TurboQuant KV 캐시, 연속 배칭
- oQe: Hessian 기반 GPTQ — 동일 비트에서 더 나은 양자화 품질
- MoE 배치 처리 15x 가속 (256 experts × 40 layers 동시 처리)

### 3.5 llama.cpp (범용 대안)

- expert offloading: `-ot "exps=CPU"` — MoE expert를 CPU/SSD로 오프로드
- GGUF 포맷 지원 (Unsloth Dynamic 2.0 양자화)
- 안정성 높지만 Flash-MoE 대비 최적화 부족

## 4. 적용할 최적화 전략 (우선순위순)

### 전략 1 — Q3 GGUF Experts [즉시 적용]
- **효과**: 4-bit 대비 36% 빠른 디코딩, 더 작은 파일 → 캐시 히트율 향상
- **품질**: PPL 3.81 vs 3.64 (4-bit) — 거의 동일
- **구현**: gorroai 포크의 `--q3-experts` 플래그

### 전략 2 — Temporal Expert Prediction [즉시 적용]
- **효과**: +55% 속도 (단독 최대 기여)
- **원리**: 27% 토큰 간 라우팅 상관관계를 이용한 다음 레이어 expert prefetch
- **구현**: gorroai 포크의 `--predict` 플래그
- **주의**: M5 base는 M5 Max보다 메모리 대역폭이 낮으므로 효과가 다를 수 있음. 테스트 필요

### 전략 3 — cache-io-split [즉시 적용]
- **효과**: SSD I/O 효율 향상
- **구현**: `--cache-io-split 4`

### 전략 4 — K=3 Expert 프루닝 [실험적]
- **효과**: SSD 읽기량 25% 감소 (K=4 → K=3)
- **품질 리스크**: K=4까지는 품질 저하 없음으로 확인됨. K=3에서 "가장 큰 품질 드롭" 발생한다는 보고
- **구현**: 코드 수정 필요 (expert 선택 로직의 top-K 값 변경)
- **32GB 환경에서 특히 유효**: 페이지 캐시가 제한적이므로 읽기량 감소가 직접적 속도 향상

### 전략 5 — ML 기반 캐시 교체 정책 [고급]
- **논문**: FlashMoE (arxiv 2601.17063)
- **효과**: LRU 대비 캐시 히트율 51% 향상, 2.6x 속도 향상
- **원리**: LRU가 퇴출한 expert의 34.2%가 5스텝 내 재사용됨. ML 모델이 recency + frequency 신호를 결합하여 최적 퇴출 결정
- **구현 난이도**: 높음 — OS 페이지 캐시 대신 사용자 공간 캐시 구현 필요
- **주의**: 원본 Flash-MoE에서는 커스텀 캐시가 오히려 느렸음. ML 캐시는 단순 LRU/LFU와 다르지만, Apple Silicon에서의 검증 필요

### 전략 6 — HOBBIT 스타일 캐시 미스 대체 [실험적]
- **효과**: 캐시 미스 시 SSD 읽기 자체를 건너뜀
- **원리**: 미스된 expert를 저정밀 버전(이미 캐시에 있는 유사 expert)으로 대체
- **품질 리스크**: 미미하지만 측정 필요

### 전략 7 — Same-layer Expert Prediction [학술 최신]
- **효과**: cross-layer prediction보다 더 정확한 expert 예측 (93-95%)
- **원리**: 이전 레이어 출력 대신 같은 레이어의 pre-attention 가중치로 expert를 예측
- **구현 난이도**: 매우 높음 — 아키텍처 수정 필요

### 전략 8 — TurboQuant KV Cache [메모리 절약]
- **효과**: KV 캐시를 즉시 양자화하여 피크 메모리 감소
- **용도**: 더 긴 컨텍스트 사용 가능 또는 페이지 캐시에 더 많은 공간 확보
- **참고**: omlx 프로젝트에 구현 있음

## 5. 32GB M5 예상 성능

### 이론적 최대값 계산

```
SSD 읽기: 6.3 GB/s
Q3 expert 크기: ~5MB
토큰당 SSD 읽기: 5MB × K=4 × 60 layers = ~1.2GB (캐시 미스 시)
이론 최대 (캐시 미스 100%): 6.3 / 1.2 ≈ 5.25 tok/s
캐시 히트 50% 가정: 유효 읽기 0.6GB → ~10.5 tok/s 이론 상한
Temporal prediction으로 I/O 은닉: 추가 개선
```

### 현실적 예상치

| 설정 | 예상 속도 |
|---|---|
| 원본 Flash-MoE (4-bit, no optimization) | 0.8-1.5 tok/s |
| 원본 Flash-MoE (2-bit) | 1.5-2.5 tok/s |
| gorroai Q3 GGUF + predict + cache-io-split | **3-5 tok/s** |
| + K=3 프루닝 | **4-6 tok/s** |
| + ML 캐시까지 | **5-7 tok/s** |

→ 모든 최적화 적용 시 원본 Flash-MoE M3 Max 48GB (4.4 tok/s)에 근접하거나 초과 가능

## 6. 양자화 vs 품질 — 실제 벤치마크

GPT-3 급이라는 주장은 **완전히 거짓**. 실측 데이터:

| 모델/설정 | MMLU | GPQA Diamond | 등급 |
|---|---|---|---|
| GPT-3 (175B) | ~43% | - | 구시대 |
| GPT-3.5 | ~70% | ~28% | 구세대 |
| GPT-4 | ~86% | ~39% | 이전 세대 |
| **397B 원본 (FP16)** | **~89%** | **~65%** | **GPT-5.2급** |
| **397B Q4_K_XL** | 원본 -1점 이내 | 거의 동일 | GPT-5.2급 |
| **397B Q3_K_XL** | 원본 -1점 이내 | 거의 동일 | GPT-5.2급 |
| **397B 2.46BPW** | **87.86%** | **82.32%** | GPT-4o+급 |

### 양자화 주의사항
- Unsloth Dynamic 2.0: 중요 레이어를 8/16-bit로 업캐스트하여 품질 유지
- 2-bit 양자화 시 JSON 출력 깨짐 (`\name\` vs `"name"`) → tool calling 불가
- ssm_out 텐서는 양자화에 매우 민감 — 최소 4-bit 유지 권장
- ffn_down_exps 양자화도 주의 필요

## 7. 구현 로드맵

### Phase 1: 기본 구동 확인 (Day 1)
1. gorroai/flash-moe 클론 및 빌드

3. TurboQuant KV 캐시 적용

1. Open WebUI 또는 llama.cpp 호환 API 서버 연동

## 9. 실패한 접근법 (피해야 할 것들)

원본 Flash-MoE의 58개 실패 실험 + 커뮤니티 보고에서 확인된 것들:

1. **LZ4 압축**: 해제 오버헤드가 SSD 읽기 절약보다 큼 (-13%)
2. **Custom Metal LRU 캐시**: OS 페이지 캐시와 경쟁하여 오히려 느림 (-38%)
3. **Speculative expert prefetching (무조건적)**: GPU contention으로 대참사 (-73%)
4. **SSD DMA + GPU compute 동시 실행**: 통합 메모리 버스 경합으로 레이턴시 스파이크
5. **MXFP4 양자화**: Q4_K보다 많은 텐서에서 열등 (4.25 vs 4.5 BPW인데 품질 더 낮음)
6. **ssm_out 2-bit 양자화**: KLD 급격히 악화
7. **attn_* 텐서 공격적 양자화**: 하이브리드 아키텍처에서 특히 민감
