## 0. P21 의견서에 대한 반론

P21 의견서(`20260415_092844_P22_Flash_MOE_의견서.md`)는 **Flash-MoE SSD 스트리밍 기술을 전혀 모른 상태**에서 작성되었다. 이로 인해 "32GB에서 120B+ 불가"라는 결론에 도달했으나, 이는 **잘못된 전제에 기반한 오판**이다.

### Flash-MoE가 뒤집는 것

| P21 의견서 주장 | Flash-MoE 현실 |
|---|---|
| "32GB에서 120B+ MoE 구동 불가" | **122B, 397B 모두 구동 가능** (RAM 5.5GB만 사용, 나머지 SSD 스트리밍) |
| "60-70% 규칙: 모델은 RAM의 60-70% 이하" | Flash-MoE는 모델 크기와 무관하게 **~6GB RAM만 상주** |
| "MoE + ComfyUI 동시 운영 불가 (33GB > 32GB)" | Flash-MoE(6GB) + ComfyUI(15GB) = **21GB → 동시 가능** |
| "핫스왑이 유일한 해법" | Flash-MoE는 핫스왑 없이도 ComfyUI와 **동시 운영 가능** |
| "하드웨어 근본 해결은 128GB Mac 뿐" | 32GB에서도 Flash-MoE로 **GPT-5급 모델 구동 가능** |

### Flash-MoE 핵심 원리 (30초 요약)

```
[기존 방식]   모델 전체(209GB)를 RAM에 로드 → 32GB에서 불가능

[Flash-MoE]   비전문가 가중치(5.5GB)만 RAM에 상주
              토큰 1개 생성 시 → 512개 expert 중 4개만 SSD에서 실시간 로드
              각 expert ~5MB × 4개 × 60 layers = ~1.2GB/토큰 (SSD 읽기)
              SSD 6.3 GB/s + macOS 페이지 캐시(71% 히트) → 실용적 속도 달성
```

### P21 의견서에서 여전히 유효한 부분

- **토큰 절감 추정 (~44%)**: 방향성 유효, 122B 기준으로 절감률 더 높아질 수 있음
- **Worker 위임 확장 계획**: 위임 범위 확대 방향 그대로 채택
- **워크로드 스케줄링 자동화**: 핫스왑 코드 구조 참고 가능
- **Speculative Decoding**: Draft+Verifier 구조 유효

## 2. 모델 포트폴리오 전략

### 2-1. 텍스트 LLM (목표 1)

> **P21 오류 정정**: "30B-A3B가 상한"이 아니라, Flash-MoE로 122B/397B도 구동 가능

| 역할 | 모델 | 총 파라미터 | 활성 | 디스크 | RAM 사용 | 실행 방식 | 예상 속도 | 벤치마크 |
|---|---|---|---|---|---|---|---|---|
| **메인 추론** | Qwen3.5-122B-A10B | 122B | 10B | ~65GB (Q4) | **~6GB** | Flash-MoE | 5-10 tok/s | GPT-4o+급 |
| **빠른 응답** | Qwen3.5-35B-A3B | 35B | 3B | ~19GB (Q4) | ~19GB | RAM 적재 | 15-30 tok/s | GPT-4 mini급 |
| **최대 품질** | Qwen3.5-397B-A17B | 397B | 17B | ~150GB (Q3) | **~6GB** | Flash-MoE | 1.5-3 tok/s | GPT-5급 |

**왜 122B가 메인인가?**
- 397B (150GB Q3): 페이지 캐시 26GB로 캐시 히트율 ~17% → SSD I/O 병목 극심
- 122B (65GB Q4): 페이지 캐시 26GB로 캐시 히트율 **~40%** → 397B 대비 2배 이상 빠름
- 35B (19GB Q4): RAM 완전 적재 → 가장 빠르지만 벤치마크 낮음

## 3. 성능 예상 (M5 base 153 GB/s 반영)

> **주의**: gorroai의 20.34 tok/s는 **M5 Max 128GB** (614 GB/s, 40코어) 기준.
> M5 base(153 GB/s, 10코어)는 대역폭 1/4, GPU 1/4이므로 직접 외삽 불가.

### 3-1. 텍스트 LLM

| 모델 | 설정 | 예상 속도 | 비고 |
|---|---|---|---|
| **122B (Q4)** | Flash-MoE + predict + cache-io-split | **5-10 tok/s** | 메인 추론, 실용적 |
| 35B (Q4) | RAM 적재 (Ollama+MLX) | **15-30 tok/s** | 빠른 응답용 |
| 397B (Q3) | Flash-MoE 풀 최적화 | **1.5-3 tok/s** | 최대 품질, 느림 |
| 397B (Q3) | + K=3 프루닝 | **2-4 tok/s** | 실험적 |

### 3-2. 동시 운영 메모리 예산

```
[Flash-MoE 모델] ← RAM ~6GB (비전문가 가중치만)
+ [ComfyUI FLUX]  ← RAM ~15GB (FP8 모델 로드)
+ [macOS]         ← RAM ~6GB
────────────────────────────
합계:              ~27GB / 32GB  → ✅ 동시 가능! (5GB 여유)
```

→ **P21 의견서 v1의 "동시 불가" 판정을 뒤집음.** Flash-MoE 모델은 RAM 6GB만 차지하므로 ComfyUI와 동시 운영이 가능하다.

### 3-3. ⚠️ 페이지 캐시 경합 (P21 v2 피드백 반영)

> **P21 지적**: 동시 운영은 메모리상 가능하나, **페이지 캐시 경합**으로 Flash-MoE 속도가 대폭 하락할 수 있다.

```
[Flash-MoE 단독]
사용 가능 페이지 캐시: 32GB - 6GB(macOS) - 6GB(Flash-MoE) = ~20GB
122B Q4 65GB 중 20GB 캐시 → 히트율 ~31%
→ 5-10 tok/s 달성 가능

[Flash-MoE + ComfyUI 동시]
사용 가능 페이지 캐시: 32GB - 6GB(macOS) - 6GB(Flash-MoE) - 15GB(ComfyUI) = ~5GB
122B Q4 65GB 중 5GB 캐시 → 히트율 ~8%
→ 거의 모든 expert를 SSD에서 읽어야 함 → 속도 대폭 저하 (2-4 tok/s 추정)
```

### 3-4. 하이브리드 운영 전략 (P21 v2 제안 채택)

페이지 캐시 경합 문제에 대한 대응:

```
[이미지 생성 Phase] — ComfyUI 활성 상태
  └── LLM: 35B-A3B RAM 적재 (19GB) 사용
      → ComfyUI(15GB) + 35B(19GB) + macOS(6GB) = 40GB → ❌ 초과
      → 대안: ComfyUI 생성 완료 대기 후 35B 사용, 또는 Flash-MoE 저속(2-4 tok/s) 감수

[비이미지 Phase] — ComfyUI 비활성
  └── LLM: 122B Flash-MoE (6GB) 사용
      → 캐시 ~20GB → 히트율 ~31% → 5-10 tok/s

[결론]
  · ComfyUI 이미지 생성 중 LLM 동시 사용: Flash-MoE 122B (저속 2-4 tok/s 감수)
  · ComfyUI 미사용 시: Flash-MoE 122B (정상 속도 5-10 tok/s)
  · 인터랙티브 빠른 응답 필요 시: ComfyUI 종료 후 35B RAM 적재 (15-30 tok/s)
```

### 3-5. ⚠️ SSD I/O 경합 (P21 v2 피드백 반영)

Flash-MoE의 SSD 스트리밍과 ComfyUI의 SSD 접근이 동시 발생 시:
- Flash-MoE: 토큰당 ~1.2GB SSD 읽기
- ComfyUI: 모델 로드 시 ~15GB + 샘플링 중 간헐적 접근
- **SSD 6.3 GB/s 대역폭을 분할** → Flash-MoE 속도 추가 저하 가능

→ **Phase 2에서 `iostat` 벤치마크로 실측 필수**

## 4. Claude 연동 아키텍처

### 4-1. 전체 구조

```
[Claude Code / Claude Desktop]
       │ MCP Protocol
       ▼
[FlashMOE MCP Server] (Python, localhost:5050)
  ├── /text        → Ollama API → Qwen3.5-122B (Flash-MoE) or 35B (RAM)
  ├── /image-gen   → ComfyUI API → FLUX 2.0 (빠른) or HunyuanImage (품질)
  ├── /audio       → Whisper API → whisper-large-v3
  └── /video       → MLX-VLM API → Qwen3-VL-30B-A3B
```

### 4-2. Worker 위임 확장

현재 Ollama Worker (7B/14B) → **Qwen3.5-122B Flash-MoE로 교체**
- 기존 위임: 요약/추출 (사실 확인)
- 확장 위임: + 구조화 생성(JSON/manifest), 1차 코드 리뷰, 번역, 파일 분석/파싱
- 예상 API 토큰 절감: **~55%** (P21 v2 재추정 채택 — v1의 44%에서 상향, 122B 위임 범위 확대 반영)

### 4-3. 라우팅 미들웨어

```
[작업 요청] → [복잡도 판단기]
  ├── 단순 (토큰 <2K, 구조화 출력) → 로컬 MoE
  ├── 중간 (코드 리뷰, 번역)        → 로컬 MoE + 10% 품질 샘플링
  └── 복잡 (아키텍처 설계, 각본)     → Claude API
```

## 7. Flash-MoE 멀티모달 적용성

### Flash-MoE가 적용 가능한 조건

Flash-MoE는 **MoE 아키텍처(sparse experts)인 모델에만 적용 가능**하다. Dense 모델에는 적용 불가.

| 모델 | MoE 여부 | Flash-MoE 적합도 | 이유 |
|---|---|---|---|
| Qwen3.5 텍스트 LLM (122B/397B) | MoE (512 experts) | **최적** | 99.2% 비활성, 검증됨 |
| **HunyuanImage-3.0** | **MoE (64 experts)** | **최적** | 87.5% 비활성, 오토리그레시브 = 텍스트 LLM과 동일 |
| Qwen3-VL-30B-A3B | MoE | **양호** | 모달리티별 라우팅 차이 주의 |
| FLUX / Stable Diffusion | **Dense** | **불가** | MoE 아님, 전체 파라미터 활성 |
| Whisper | **Dense** | **불가** | MoE 아님, 3GB로 RAM 충분 |

### HunyuanImage-3.0이 특별한 이유

전통 디퓨전 모델(FLUX, SD)은 20-50 디노이징 스텝을 반복하며 모든 파라미터를 활성화한다. MoE가 아니므로 Flash-MoE 적용 불가.

HunyuanImage-3.0은:
1. **MoE 아키텍처**: 64 experts 중 8개만 활성 (87.5% 비활성)
2. **오토리그레시브 방식**: 디퓨전이 아닌 토큰 단위 생성 → Flash-MoE 프리페치 그대로 적용
3. **80B 파라미터**: 50억 이미지-텍스트 쌍으로 학습한 오픈소스 최대 이미지 생성 모델
4. **RAM 6GB**: Flash-MoE로 비전문가 가중치만 상주 → ComfyUI FLUX와 동시 운영 가능

**현재 멀티모달 Flash-MoE 구현체는 존재하지 않음** (2026-04 기준) → 선구적 연구 기회
