## 0. v1 의견서 오류 정정

v1 의견서(`20260415_092844_`)는 **Flash-MoE SSD 스트리밍 기술을 인지하지 못한 상태**에서 작성되어 핵심 결론이 잘못되었다. P22 계획안의 반론을 수용하며 아래와 같이 정정한다.

| v1 결론 (잘못됨) | v2 정정 |
|---|---|
| "32GB에서 120B+ MoE 구동 불가" | Flash-MoE로 122B/397B 모두 구동 가능 (RAM ~6GB, SSD 스트리밍) |
| "MoE + ComfyUI 동시 운영 불가 (33GB > 32GB)" | Flash-MoE(6GB) + ComfyUI(15GB) = 21GB → **동시 가능** |

| "하드웨어 근본 해결은 128GB Mac 뿐" | 32GB에서도 GPT-5급 모델 실행 가능 |
| "Qwen3-30B-A3B가 실용적 상한" | Qwen3.5-122B-A10B가 메인, 397B도 가능 |

**v1에서 여전히 유효한 부분**:

### 1-3. 실전에서 겪은 핵심 제약 (TRAP 시스템, 30건)

P21은 제작 과정에서 겪은 시행착오를 TRAP-NNN 번호로 관리하고 있다 (총 30건). P22와 직접 관련되는 항목:

| TRAP | 제목 | P22 관련성 |
|---|---|---|
| **TRAP-019** | LP+ComfyUI 동시 실행 시 메모리 부족 | **Flash-MoE가 해결 가능** — 6GB만 사용하므로 ComfyUI와 동시 가능 |
| **TRAP-023** | ComfyUI 로드 시 free 90%→37% | Flash-MoE 도입 시 ComfyUI(15GB) + Flash-MoE(6GB) = 21GB, 남은 11GB로 충분 |
| **TRAP-024** | Ollama 14B Worker 품질 한계 | **122B-A10B 교체로 근본 해결** — 위임 범위 대폭 확대 가능 |
| **TRAP-016~020** | LivePortrait 얼굴 미감지/모션 제한 | LLM과 무관, P22 범위 밖 (LP 자체 한계) |
| **TRAP-001~006** | AI 일러스트 텍스트 누출/팔 절단 등 | ComfyUI FLUX 한계 → HunyuanImage-3.0가 대안될 수 있음 |

### 1-4. P21이 가장 아쉬운 3가지

1. **Ollama Worker 품질 부족**: 7~14B 모델로는 "사실 확인(요약/추출)"만 위임 가능. JSON 생성, 코드 리뷰, 번역은 품질이 부족하여 Claude API에 의존 → 토큰 소비 과다
2. **ComfyUI ↔ LP ↔ Ollama 동시 실행 불가**: Phase마다 수동 전환 필요, 전환 오버헤드 15~30초 × 수십 회/세션
3. **일러스트 품질 천장**: FLUX Kontext Dev는 텍스트 누출, 캐릭터 일관성 문제가 상존 — 더 큰 모델(HunyuanImage-3.0 80B)이 해결할 수 있을 가능성

## 2. P22 계획안 교차 검토 — P21 실전 관점

### 2-1. Flash-MoE 동시 운영: P21이 확인해주는 것

P22 계획안의 핵심 주장 — "Flash-MoE(6GB) + ComfyUI(15GB) = 21GB → 동시 가능"을 P21 실측 데이터로 교차 검증한다.

```
[P21 실측 기준]
macOS 기본:        ~6GB (실측)
ComfyUI FLUX Q8:   ~15GB (실측, TRAP-023 기록)
Flash-MoE 122B:    ~6GB (P22 추정, gorroai 벤치마크 기반)
────────────────────────────
합산:               ~27GB / 32GB → ✅ 5GB 여유

[비교: 현재 P21 불가능한 조합]
macOS(6GB) + ComfyUI(15GB) + Ollama 14B(10GB) = 31GB → ❌ 빡빡
macOS(6GB) + ComfyUI(15GB) + LP(6GB) = 27GB → ⚠️ 가능하나 불안정
```

**P21의 검증 포인트**: 
- ComfyUI의 15GB는 **모델 로드 즉시** 점유되며 생성 중에도 해제되지 않음 (TRAP-023)
- Flash-MoE가 실제로 6GB만 사용한다면, ComfyUI와 동시 운영은 메모리상 가능
- 단, **SSD I/O 경합** 가능성은 검증 필요: ComfyUI도 모델 로드/샘플링 시 SSD를 사용하므로, Flash-MoE의 SSD 스트리밍과 동시에 발생하면 속도 저하 가능

### 2-2. 122B-A10B의 Worker 위임 확장: P21이 원하는 것

P21의 Ollama Worker 위임 현황과 122B 교체 시 확장 가능 범위:

| 작업 유형 | 현재 (14B) | 122B-A10B 교체 시 | 이유 |
|---|---|---|---|
| **요약/추출** | ✅ 위임 중 | ✅ 품질 대폭 향상 | 기존에도 가능, 정확도↑ |

## 3. P22 계획안에 대한 P21 의견

### 3-1. 동의하는 부분

| P22 주장 | P21 동의 근거 |
|---|---|
| Flash-MoE로 ComfyUI와 동시 운영 가능 | P21 실측: ComfyUI 15GB + macOS 6GB = 21GB. Flash-MoE 6GB 추가해도 27GB. |
| 122B-A10B가 메인 추론 모델로 적합 | P21 Worker 14B의 한계를 절감. 122B면 위임 범위 대폭 확장. |
| Open WebUI + Tailscale 아이폰 접근 | P21에서 ComfyUI를 8188 포트로 원격 접근 중. 동일 인프라 재사용 가능. |
| Phase 0~3 로드맵 (Day 1~7) | 빠른 검증 → 점진 확대 전략 적절. |

### 3-2. 주의가 필요한 부분

#### A. SSD I/O 경합 위험

P22 계획안은 Flash-MoE의 SSD 읽기와 ComfyUI의 SSD 접근이 동시에 발생하는 상황을 고려하지 않았다.

- Flash-MoE: 토큰 1개당 ~1.2GB SSD 읽기 (4 experts × 5MB × 60 layers)
- ComfyUI: 모델 로드 시 ~15GB SSD 읽기 + 샘플링 중 간헐적 SSD 접근
- **동시 발생 시**: SSD 6.3 GB/s 대역폭을 분할 → Flash-MoE 속도 저하 가능

**P21 권장**: Phase 2에서 "ComfyUI 동시 운영 테스트" 항목에 **SSD I/O 벤치마크** 추가. `iostat`으로 실측하여 병목 확인.

#### B. 페이지 캐시 경합

P22 계획안에서 122B Q4(65GB)의 페이지 캐시 히트율을 ~40%로 추정했다. 그러나 ComfyUI가 15GB를 점유하면 OS가 사용할 수 있는 페이지 캐시 용량이 줄어든다.

```
[Flash-MoE 단독]
사용 가능 페이지 캐시: 32GB - 6GB(macOS) - 6GB(Flash-MoE) = ~20GB
122B Q4 65GB 중 20GB 캐시 = 히트율 ~31%

[Flash-MoE + ComfyUI]
사용 가능 페이지 캐시: 32GB - 6GB(macOS) - 6GB(Flash-MoE) - 15GB(ComfyUI) = ~5GB
122B Q4 65GB 중 5GB 캐시 = 히트율 ~8%
→ 거의 모든 expert를 SSD에서 읽어야 함 → 속도 대폭 저하
```

**P21 권장**: ComfyUI 동시 운영 시 122B보다 **35B-A3B RAM 적재**가 더 현실적일 수 있다. 이미지 생성 Phase에서는 35B(19GB, RAM 완전 적재)를 사용하고, 이미지 생성 없는 Phase에서만 122B Flash-MoE를 사용하는 **하이브리드 전략** 검토.

#### C. 5~10 tok/s의 실용성 (M5 base)

P22 계획안의 122B 예상 속도 5~10 tok/s는 M5 base(153 GB/s)에서의 추정이다.

- P21의 Ollama 14B: ~30 tok/s → 실시간 대화 수준
- 122B Flash-MoE: 5~10 tok/s → **응답 1000토큰에 100~200초**
- Claude API: ~50~80 tok/s (스트리밍) → 비교 불가

**P21 견해**: Worker 위임 작업(요약, JSON 생성, 파싱)은 비동기 배치이므로 5~10 tok/s도 충분하다. 단, **인터랙티브 대화**에는 35B-A3B(15~30 tok/s)가 적합. P22의 "메인 추론 122B + 빠른 응답 35B" 이원 구조에 동의.

#### D. 397B의 실용성 의문

P22 계획안에 397B(Q3, 150GB)가 "최대 품질 모드"로 포함되어 있으나:

- 예상 속도: 1.5~3 tok/s → 응답 1000토큰에 333~667초 (5~11분)
- SSD 150GB 점유 → 전체 모델 예산 319GB 중 47%
- 페이지 캐시 히트율 ~17% → SSD I/O 병목 극심

**P21 권장**: Phase 6(Day 22~23)에서 벤치마크 후 판단. 1.5 tok/s 미만이면 **폐기하고 122B에 집중**하는 것이 SSD 용량 효율적. 150GB를 절약하면 다른 멀티모달 모델(HunyuanImage 42GB + Qwen3-VL 16GB)에 여유를 줄 수 있다.

### 3-3. P21이 P22에 요청하는 것

1. **Worker 위임 인터페이스 표준화**: P21의 `worker_rules.md`와 호환되는 API 규격 확정. Claude Code에서 `ollama run` 명령어로 직접 호출하고 있는 현재 방식을 MCP 서버 경유로 전환할 때 P21 파이프라인이 깨지지 않도록.

2. **ComfyUI 연동 MCP 엔드포인트**: P22 Phase 4의 `/image-gen` 엔드포인트가 P21의 ComfyUI 워크플로우(FLUX Kontext Dev GGUF Q8 + UnetLoaderGGUF 노드)와 호환되어야 한다. P21의 `generate_scenes.py`가 직접 ComfyUI API를 호출하고 있으므로, MCP 래퍼가 기존 워크플로우 JSON을 그대로 전달할 수 있어야 함.

3. **HunyuanImage-3.0 A/B 테스트**: Phase 5에서 P21 EP.01의 실제 씬 프롬프트로 FLUX vs HunyuanImage 비교. P21이 프롬프트 셋(로케이션 8장 + 캐릭터 24종)을 제공할 수 있다.

4. **핫스왑 → 동시 운영 전환 검증**: Phase 2 결과에 따라, P21의 현재 "ComfyUI 종료 → LP 실행 → LP 종료 → ComfyUI 실행" 순차 패턴을 "Flash-MoE + ComfyUI 동시 + LP 순차"로 개선할 수 있는지 확인.

## 5. P21 멀티모달 도구별 P22 연동 방안

### 5-1. ComfyUI + FLUX (이미지 생성)

| 항목 | 현재 P21 | P22 도입 후 |
|---|---|---|
| **모델** | FLUX Kontext Dev GGUF Q8 (~12GB) | FLUX 유지 + HunyuanImage-3.0 (Phase 5 검증 후) |
| **호출 방식** | Python → ComfyUI REST API (localhost:8188) | MCP `/image-gen` → ComfyUI API (동일) |
| **RAM** | ~15GB | ~15GB (FLUX) 또는 ~6GB (HunyuanImage Flash-MoE) |
| **동시 운영** | ❌ Ollama Worker와 동시 불가 | ✅ Flash-MoE(6GB)와 동시 가능 |
| **워크플로우** | `generate_scenes.py` → ComfyUI prompt JSON | MCP 래퍼가 동일 JSON 전달 (호환 필수) |

**P21 주의사항**: 
- ComfyUI 워크플로우는 `UnetLoaderGGUF` 노드를 사용하며, 이는 ComfyUI Manager로 설치한 커스텀 노드. MCP 래퍼가 이를 인지해야 함.
- CLIP 모델: `t5xxl_fp8_e4m3fn_scaled.safetensors` — FP8 모델은 MPS에서 사용 불가(GGUF Q8 사용 중).

### 5-2. LivePortrait (캐릭터 애니메이션)

| 항목 | 현재 P21 | P22 도입 후 |
|---|---|---|
| **모델** | LivePortrait pretrained_weights (~2.1GB) | 변동 없음 (LLM 무관) |
| **호출 방식** | Python `inference.py` (PYTORCH_ENABLE_MPS_FALLBACK=1) | 변동 없음 |
| **RAM** | 피크 ~6GB, 상시 ~4GB | 동일 |
| **P22 관련** | LP + Flash-MoE 동시 가능 (6+4+6=16GB) | ✅ 여유 충분 |

**P21 의견**: LivePortrait는 P22의 영향을 거의 받지 않음. 다만 LP 실행 중에 Flash-MoE 122B를 동시에 돌릴 수 있다면, LP 배치 중 다른 LLM 작업을 병행할 수 있어 효율적.

### 5-3. TTS (edge-tts + Typecast)

| 항목 | 현재 P21 | P22 도입 후 |
|---|---|---|
| **edge-tts** | CPU 경량, <0.5GB | 변동 없음 |
| **Typecast** | 외부 API (PRO 플랜 ₩35,000/월) | 변동 없음 |
| **P22 관련** | Whisper (음성→텍스트) 도입 시 TTS ↔ STT 루프 가능 | 검토 가치 있음 |

**P21 의견**: P22의 Whisper 도입이 P21에 도움될 수 있는 시나리오 — Typecast TTS 출력물을 Whisper로 전사하여 각본 대비 자동 QA (발음 오류, 누락 검출).

### 5-4. FFmpeg + Python (영상 편집)

| 항목 | 현재 P21 | P22 도입 후 |
|---|---|---|
| **역할** | build_video_v3.py, chromakey, 오디오 믹싱 | 변동 없음 (LLM 무관) |
| **P22 관련** | FFmpeg 명령어 생성을 122B Worker에게 위임 가능 | 토큰 절감 기여 |
