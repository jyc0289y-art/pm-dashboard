## 1. Flash MOE 전략 분석

### 1-1. Mixture of Experts (MoE) 아키텍처란

MoE는 하나의 모델 안에 수십~수백 개의 "전문가(Expert)" 서브네트워크를 배치하되, **추론 시 입력 토큰마다 일부 전문가만 활성화**하는 아키텍처이다.

| 개념 | 설명 |
|------|------|
| **전체 파라미터** | 모델이 보유한 총 가중치 수 (예: 120B, 235B, 397B) |
| **활성 파라미터** | 토큰 1개 처리 시 실제 연산에 참여하는 파라미터 수 (예: 3B, 5B, 22B) |
| **라우터(Gating)** | 입력 토큰을 적절한 전문가에게 배분하는 경량 네트워크 |
| **희소 활성화** | 전체의 2~10%만 활성 → Dense 모델 대비 연산량 대폭 절감 |

**핵심 이점**: 전체 파라미터가 크므로 지식 용량은 Dense 대형 모델급이지만, 활성 파라미터가 작으므로 추론 연산량은 소형 모델급이다.

### 1-2. 2025~2026 주요 MoE 모델 현황

| 모델 | 전체 파라미터 | 활성 파라미터 | Expert 수 | 활성 Expert | 특징 |
|------|-------------|-------------|----------|------------|------|
| **GPT-OSS-120B** (OpenAI) | 117B | **5.1B** | 128 | 4 | MXFP4 양자화, Metal 참조 구현 제공 |
| **GPT-OSS-20B** (OpenAI) | 20B | ~2B | 32 | 4 | 16GB에서 구동 가능 |
| **Qwen3-30B-A3B** (Alibaba) | 30.5B | **3.3B** | — | — | QwQ-32B 성능, 10분의 1 활성 파라미터 |
| **Qwen3-235B-A22B** (Alibaba) | 235B | **22B** | — | — | 최상위 오픈 MoE, 128GB+ 필요 |
| **Qwen3.5-35B-A3B** (Alibaba) | 35B | **3B** | — | — | 최신, 롱컨텍스트 |
| **DeepSeek-V3** | 671B | 37B | 256 | 8 | Q5-K-M 시 ~502GB, 로컬 구동 비현실적 |
| **LLaMA-4 Maverick** (Meta) | — | — | — | — | 2025 MoE 라인업 |

### 1-3. "120B+397B" 수준의 의미

사용자가 언급한 "120B+397B"는 다음 두 가지로 해석 가능하다:

- **해석 A**: GPT-OSS-120B (전체 117B, 활성 5.1B) 같은 MoE 모델 — 파라미터 수 대비 극히 가벼운 추론
- **해석 B**: 전체 400B급 MoE (예: DeepSeek-V3 671B 계열) — 32GB Mac에서는 **구동 불가**

**현실적 타겟**: 32GB Mac 환경에서는 **GPT-OSS-120B (활성 5.1B)** 또는 **Qwen3-30B-A3B (활성 3.3B)** 수준이 상한선이다.

### 1-4. 32GB Mac에서의 실행 가능성

#### 메모리 요구량 (GGUF 양자화 기준)

| 모델 | Q4_K_M | Q8_0 | FP16 | 32GB 적합성 |
|------|--------|------|------|------------|
| **Qwen3-30B-A3B** | ~18GB | ~32GB | ~62GB | **Q4/Q6: 적합** |
| **Qwen3.5-35B-A3B** | ~20GB | ~35GB | ~70GB | **Q4: 적합** (Q8은 빡빡) |
| **GPT-OSS-20B** | ~12GB | ~20GB | ~40GB | **Q4~Q8: 여유** |
| **GPT-OSS-120B** | ~65GB+ | ~120GB+ | — | **불가** (MXFP4로도 ~40GB) |
| **Qwen3-235B-A22B** | ~130GB+ | — | — | **불가** |
| **DeepSeek-V3** | ~370GB+ | — | — | **불가** |

> **60-70% 규칙**: 모델 파일은 전체 메모리의 60~70% 이하여야 안정적. 32GB 기준 **19~22GB가 모델 상한**.

#### 추론 성능 예상 (32GB M5)

| 모델 | 양자화 | 예상 tok/s | 용도 적합성 |
|------|--------|-----------|-----------|
| **Qwen3-30B-A3B** Q4 | Q4_K_M | **33~53 tok/s** | 실시간 대화, 코드 생성 |
| **Qwen3.5-35B-A3B** Q4 | Q4_K_M | **25~40 tok/s** | 실시간 대화, 분석 |
| **GPT-OSS-20B** Q8 | Q8_0 | **20~30 tok/s** | 실시간 대화 |
| **GPT-OSS-120B** (스왑) | Q2~Q3 | **1~3 tok/s** | **비현실적** |

**결론**: 32GB M5에서 실용적으로 돌릴 수 있는 MoE 모델은 **Qwen3-30B-A3B, Qwen3.5-35B-A3B, GPT-OSS-20B** 수준이다. 120B+ 모델은 메모리 부족으로 **사실상 불가능**하다.

## 2. P21 멀티모달 대비 이점 분석

### 2-1. 현재 P21 파이프라인 구성

```
[Claude Code API] ──(주력 에이전트)──→ 코드 생성/수정, 각본 검토, 아키텍처 설계
     │
     ├── [Ollama Worker] ──→ 요약/추출 위임 (사실 확인 작업)
     ├── [ComfyUI + FLUX GGUF] ──→ 이미지 생성 (~10-15GB)
     ├── [LivePortrait] ──→ 캐릭터 애니메이션 (~4-6GB)
     ├── [edge-tts] ──→ TTS 초안 (CPU, 경량)
     ├── [FFmpeg + Python] ──→ 영상 편집 (CPU)
     └── [Typecast API] ──→ TTS 최종 (외부 서비스)
```

### 2-2. Flash MOE 로컬 모델 도입 시 이점 있는 작업

| 작업 | 현재 방식 | MoE 대체 시 이점 | 이점 크기 |
|------|----------|----------------|----------|
| **요약/추출** | Ollama (7B~14B) | 30B-A3B 수준으로 품질 대폭 향상 | ★★★★ |
| **코드 리뷰/버그 탐색** | Claude API | 1차 스크리닝을 로컬에서 → API 절감 | ★★★ |
| **각본 초안 검토** | Claude API | 문법/구조 1차 검토 로컬화 | ★★★ |
| **파일 분석/파싱** | Claude API (파일 읽기) | JSON/MD 파싱, 정합성 검사 로컬화 | ★★★ |
| **프롬프트 초안 생성** | Claude API | ComfyUI 프롬프트 초안을 로컬 생성 | ★★ |
| **번역/다국어** | Claude API | 한국어↔영어 번역 로컬화 | ★★ |
| **manifest 생성** | Claude API | 구조화된 JSON 생성 로컬화 | ★★ |

### 2-3. 이점이 없는 작업

| 작업 | 이유 |
|------|------|
| **아키텍처 설계/의사결정** | Claude Opus급 추론력 필요, 30B MoE로는 부족 |
| **복잡한 코드 생성** (build_video_v3.py 등) | 수백 줄 코드의 맥락 유지 + 정확성 필요 |
| **시네마틱 각본 집필** | 창의성 + 한국어 뉘앙스 + 심리학 정확성 동시 요구 |
| **이미지 생성** | ComfyUI/FLUX가 이미 로컬 — LLM 관여 없음 |
| **영상 편집** | FFmpeg/Python이 이미 로컬 — LLM 관여 없음 |
| **LivePortrait** | 전용 모델 — LLM과 무관 |
| **TTS** | edge-tts/Typecast 전용 — LLM과 무관 |

### 2-4. 이점 극대화를 위한 보완사항

1. **Worker 위임 규칙 고도화**: 현재 `worker_rules.md`의 위임 기준을 MoE 모델 성능에 맞게 재설정
   - 7B→30B-A3B 전환 시 위임 가능 작업 범위 확대
   - "사실 확인"뿐 아니라 "구조화된 생성"(JSON, manifest)까지 위임 가능
2. **Speculative Decoding 활용**: 소형 모델(Draft)로 초안 → MoE 모델(Verifier)로 검증
3. **프롬프트 캐싱**: 반복되는 시스템 프롬프트를 KV 캐시로 재사용

2. **품질 검증 루프**: 로컬 MoE 출력을 Claude API가 spot-check (전수 검증 X, 10% 샘플링)
3. **점진적 전환**: 7일간 병렬 실행(같은 작업을 MoE+API 양쪽에서) → 품질 갭 측정 후 전환 확대

## 4. 멀티 에이전트 운영 가능성

### 4-1. 워크로드별 메모리 요구량

| 워크로드 | 피크 메모리 | 상시 메모리 | 비고 |
|---------|-----------|-----------|------|
| **macOS + 기본 앱** | — | ~6GB | 항상 점유 |
| **ComfyUI + FLUX Q8** | ~15GB | ~15GB | 모델 로드 시 free 90%→37% (TRAP-023) |
| **LivePortrait** | ~6GB | ~4GB | MPS 가속 |
| **Ollama (30B-A3B Q4)** | ~20GB | ~18GB | KV 캐시 포함 |
| **FFmpeg/Python** | ~2GB | ~1GB | CPU 위주 |
| **edge-tts** | <0.5GB | <0.5GB | CPU 경량 |

### 4-2. 동시 운영 시나리오 분석

#### 시나리오 A: MoE + ComfyUI (동시)
```
macOS(6GB) + FLUX(15GB) + MoE-30B(18GB) = 39GB → ❌ 32GB 초과 (7GB 부족)
```

#### 시나리오 B: MoE + LivePortrait (동시)
```
macOS(6GB) + LP(4GB) + MoE-30B(18GB) = 28GB → ⚠️ 가능하나 KV 캐시 여유 없음
```

#### 시나리오 C: MoE 단독 (ComfyUI/LP 미실행)
```
macOS(6GB) + MoE-30B(18GB) = 24GB → ✅ 8GB 여유 (KV 캐시 + 기타)
```

#### 시나리오 D: ComfyUI + LP (기존, MoE 없이)
```
macOS(6GB) + FLUX(15GB) = 21GB → ✅ (LP는 순차 전환)
```

### 4-3. 결론: 동시 운영은 불가능

32GB 통합 메모리에서 **MoE 30B + ComfyUI FLUX를 동시에 올리는 것은 물리적으로 불가능**하다. MoE + LivePortrait도 빡빡하여 실용적이지 않다.

**현재 P21도 ComfyUI와 LivePortrait를 동시 실행하지 못하고 순차 전환하고 있다** (TRAP-019). MoE를 추가하면 3자 순차 전환이 된다.

### 4-4. 소프트웨어 보강책

#### A. 모델 핫스왑 (가장 현실적)

```
[Phase: 이미지 생성]     → ComfyUI FLUX 로드, MoE 언로드
[Phase: 코드/분석 작업]   → MoE 로드, ComfyUI 종료
[Phase: 캐릭터 애니]      → LivePortrait 로드, MoE 언로드
```

- **장점**: 추가 비용 0원, 소프트웨어만으로 해결
- **단점**: 모델 로드/언로드 시간 (FLUX ~30초, MoE ~20초, LP ~15초)
- **구현**: `ComfyUI 생명주기 자동 관리` (TRAP-023 해결책)를 MoE까지 확장
  - 작업 완료 즉시 모델 언로드 → 다음 워크로드에 메모리 반환
  - Ollama의 `OLLAMA_KEEP_ALIVE=0` 설정으로 idle 시 즉시 언로드

#### B. 양자화 레벨 다운 (Q4 → Q3/Q2)

| 양자화 | 모델 크기 | 품질 영향 | 메모리 절감 |
|--------|---------|---------|-----------|
| Q4_K_M | ~18GB | 기준선 | — |
| Q3_K_M | ~14GB | 약간 저하 | -4GB |
| Q2_K | ~11GB | 눈에 띄는 저하 | -7GB |
| IQ2_XS | ~9GB | 상당한 저하 | -9GB |

Q3_K_M(14GB)이면 ComfyUI(15GB)와의 합산이 35GB → 여전히 초과하지만, FLUX를 Q4로 내리면 근접.

#### C. Speculative Decoding

- **Draft 모델** (Qwen3-0.6B, ~1GB) + **Verifier** (Qwen3-30B-A3B Q4, ~18GB)
- Draft가 5~8 토큰 선생성 → Verifier가 일괄 검증 → 2~3x 속도 향상
- 메모리 추가: ~1GB (Draft 모델 크기)
- Ollama/llama.cpp에서 지원 시작 (2025~)

#### D. 워크로드 스케줄링 자동화

```python
# 개념적 스케줄러
class WorkloadScheduler:
    def request(self, task_type):
        if task_type in ['summarize', 'parse', 'translate']:
            self.ensure_loaded('moe')      # MoE 로드, 다른 모델 언로드
        elif task_type in ['generate_image']:
            self.ensure_loaded('comfyui')   # ComfyUI 로드, MoE 언로드
        elif task_type in ['animate']:
            self.ensure_loaded('liveportrait')
```

### 4-5. 하드웨어 보강책

#### A. Mac Studio M5 Ultra (최적 업그레이드)

| 사양 | M5 (현재) | M5 Max | M5 Ultra |
|------|----------|--------|----------|
| 통합 메모리 | 32GB | 48/64/128GB | 128/192/256GB |
| 메모리 대역폭 | ~153 GB/s | ~546 GB/s | ~1092 GB/s |
| MoE 120B 구동 | ❌ | ⚠️ (128GB) | ✅ (128GB+) |
| 동시 운영 | ❌ | ⚠️ | ✅ |

- **128GB M5 Max/Ultra**: MoE 120B(Q4 ~65GB) + ComfyUI(15GB) + LP(6GB) = ~86GB → 동시 가능
- **비용**: Mac Studio M5 Ultra 128GB 예상 ~₩5,000,000~7,000,000

#### B. eGPU (Apple Silicon 비호환)

- Apple Silicon은 **eGPU를 공식 지원하지 않는다** (M1부터 Thunderbolt eGPU 지원 중단)
- 따라서 외장 GPU 옵션은 **사실상 없음**

#### C. 외장 메모리 증설

- Apple Silicon의 통합 메모리는 **SoC에 내장**되어 있어 **사후 증설 불가능**
- 유일한 방법: 더 큰 메모리의 Mac 구매

#### D. 클라우드 하이브리드 (비용 효율 고려)

| 옵션 | 비용 | 장점 | 단점 |
|------|------|------|------|

## 최종 결론

### 핵심 요약

1. **32GB M5에서 120B+ MoE는 불가능하다.** 실용적 상한은 **Qwen3-30B-A3B (활성 3.3B, 모델 ~18GB)** 또는 **GPT-OSS-20B (~12GB)** 수준이다.

2. **그럼에도 30B-A3B급 MoE는 현행 7~14B Worker 대비 큰 업그레이드다.** QwQ-32B Dense 급 성능을 1/10 연산으로 달성하므로, Worker 위임 범위를 요약/추출에서 구조화 생성, 1차 코드 리뷰, 번역까지 확장 가능하다.

3. **Claude API 토큰 ~44% 절감이 가능하다.** 파일 분석(25%), 요약(5%), JSON 생성(6%) 등 구조적 작업을 로컬화하면 주간 ~218K 토큰 절감.

4. **동시 운영은 불가능 — 핫스왑이 유일한 해법이다.** ComfyUI(15GB) + MoE(18GB) = 33GB > 32GB. Phase 단위 모델 전환으로 운영해야 한다.

5. **하드웨어 근본 해결은 128GB Mac 뿐이다.** eGPU 불가, 메모리 증설 불가. Mac Studio M5 Ultra 128GB로 업그레이드하면 120B MoE + ComfyUI + LP 동시 운영 가능.

### 우선순위 실행 계획

| 순서 | 작업 | 비용 | 효과 |
|------|------|------|------|
| 1 | Ollama Worker를 Qwen3-30B-A3B Q4로 교체 | 0원 | 품질 2~3x 향상 |
| 2 | Worker 위임 규칙 확장 (`worker_rules.md`) | 0원 | 대체율 44% 달성 |
| 3 | 모델 핫스왑 자동화 스크립트 | 0원 | Phase 전환 자동화 |
| 4 | 라우팅 미들웨어 구축 | 0원 | 수동 판단 제거 |
| 5 | (선택) Mac Studio 128GB 업그레이드 | ~₩5M+ | 동시 운영 + 120B 가능 |
